## 2. Scope

### 2.1 In (P1-2)

## 4. Output shapes

All new lookups return a shape compatible with the P1-1 `lookups[]` element — a `kind` discriminator at the top level, plus the lookup-specific fields.

### 4.1 `lookupFile`

```ts
lookupFile(intentFile, {
  symbols, topology, triage, canonicalClaims, root,
}) → {
  kind: 'file',
  intentFile: string,
  result: 'FILE_EXISTS'            // topology or symbols observes the path
        | 'NEW_FILE'               // topology complete AND doesn't list it
        | 'FILE_STATUS_UNKNOWN',   // topology absent / incomplete, no fs check available
  loc: number | null,              // LOC count when topology has it; null otherwise
  inboundFanIn: number | null,
  inboundFanInConfidence: 'grounded' | 'unavailable',
  submodule: string | null,        // via _lib/paths.mjs::buildSubmoduleResolver
  boundary: {
    status: 'ALLOWED' | 'FORBIDDEN' | 'NOT_DECLARED' | 'NOT_EVALUATED',
    rule: {                        // present only when status is ALLOWED or FORBIDDEN
      from: string, to: string, direction: 'allowed' | 'forbidden',
      declaredIn: string,
    } | null,
  },
  tags: string[],                  // e.g. ['test-only'] via isTestLikePath
  citations: string[],
}
```

Result semantics (file absence is a DIFFERENT claim from "topology doesn't list it"):

- **`FILE_EXISTS`** — `topology.nodes` has the path, OR `symbols.defIndex` has at least one identity declared in it. Either is positive evidence the file exists.
- **`NEW_FILE`** — requires ALL of: (a) `topology.json` is present, (b) `topology.meta.complete === true` (every scanned-and-parsed file listed in `nodes`), (c) the path is absent from `topology.nodes`, AND (d) the path is absent from `symbols.filesWithParseErrors` (parse-error files also exist on disk; they're just not in `nodes`). `topology.meta.complete` covers only successfully-parsed files — a file that failed to parse is real and present, just not enumerated. The P1-2 file lookup must cross-check both artifacts before claiming newness. Only one canonical field is recognized for the completeness signal: `topology.meta.complete`. No aliases.
- **`FILE_STATUS_UNKNOWN`** — topology is absent, or topology is present but completeness is unverified. Emitted with `[확인 불가, reason: ...]` citation. The advisory tells Claude "I can't tell if this file exists"; Claude decides.

**`symbols.defIndex` absence is NEVER grounded evidence for `NEW_FILE`.** A file may exist and simply export nothing (types-only file, barrel with only re-exports stripped to `{source, line}`, etc.). Using defIndex absence to claim NEW_FILE is a false-claim class per `canonical/invariants.md` §1 Iron Law.

**Boundary status semantics:**

- **`ALLOWED`** — a declared rule in `triage.json` positively matches a planned `from → to` edge the intent implies. Requires both endpoints.
- **`FORBIDDEN`** — same structural requirement; rule denies the edge.
- **`NOT_DECLARED`** — rule infrastructure consulted, no rule matches.
- **`NOT_EVALUATED`** — intent lacks planned-edge information OR triage.json absent. This is the default when the intent only names a file without declaring who imports it or what it imports. ALLOWED / FORBIDDEN may NEVER be claimed without a known `from → to` pair.

P1-2 intent schema does NOT add a `plannedEdges` field. Boundary evaluation in P1-2 is therefore LIMITED to `NOT_DECLARED` / `NOT_EVALUATED`; real ALLOWED/FORBIDDEN verdicts require planned-edge information, which is deferred to P1-3 or P2 when the post-write delta observes actual edges.

### 4.2 `lookupDependency`

```ts
lookupDependency(depName, { packageJson, symbols }) → {
  kind: 'dependency',
  depName: string,
  result: 'DEPENDENCY_AVAILABLE'
        | 'DEPENDENCY_AVAILABLE_NO_OBSERVED_IMPORTS'
        | 'NEW_PACKAGE',
  declaredIn: 'dependencies' | 'devDependencies' | 'peerDependencies' | null,
  existingImports: {
    examples: { file: string, fromSpec: string }[],   // grounded sample, capped at ~5
    observedImportCount: number | null,                // total observed count if scan can express it
    countConfidence: 'grounded' | 'sample-only',       // 'sample-only' when examples truncated or total unknown
  },
  citations: string[],
}
```

`existingImports.examples` is an explicit SAMPLE of consumers, not an exhaustive list. `observedImportCount` is the total when the lookup can count it (e.g. full `symbols.uses` iteration); `countConfidence: 'sample-only'` when only examples are available and the total is unknown.

The "unused / cleanup" framing is FORBIDDEN per docs/history/phases/p1/p1-1.md §5.5 — a package may be used by scripts/config/runtime plugins outside the static import graph.

**Package-root normalization algorithm** (applied to both the `depName` argument and every `fromSpec` observed in `symbols.uses`):

```js
function packageRoot(spec) {
  if (!spec) return null;
  // Relative / absolute paths are NOT package dependencies — skip.
  if (spec.startsWith('.') || spec.startsWith('/')) return null;
  // Scoped package: first two slash-segments.
  if (spec.startsWith('@')) {
    const parts = spec.split('/');
    if (parts.length < 2) return null;  // malformed
    return parts.slice(0, 2).join('/');
  }
  // Bare package: first slash-segment.
  return spec.split('/')[0];
}
```

Examples: `dayjs/plugin/utc` → `dayjs`; `@scope/pkg/sub/path` → `@scope/pkg`; `./relative` → `null` (excluded from dependency matching); `/abs/path` → `null`.

### 4.3 `lookupShape`

```ts
lookupShape(shapeCandidate, ctx) → {

  shape: { fields: string[] },
  result: 'UNAVAILABLE',
  citations: [
    '[확인 불가, shape-hash index not yet implemented; shape-hash fact producer lives in P4 per fact-model.md §5 producers table]',

}
```

Always the same shape. The implementation is ~10 LOC. Pinning tests (§6) prevent any heuristic field-overlap fallback from ever being added.

### 4.4 Extended `lookups[]` kinds

P1-1 emitted `lookups[]` entries with `kind: 'name'`. P1-2 appends entries with `kind ∈ {'file', 'dependency', 'shape'}`. Order in `lookups[]`: names first (stable from P1-1), then files, then dependencies, then shapes. Preserves a grep-friendly sequence for human review.

### 4.5 Extended `boundaryChecks[]`

Population rule is split by evaluation state — do NOT emit confident verdicts without endpoints.

- `boundaryStatus` ∈ `{'ALLOWED', 'FORBIDDEN'}` → populated ONLY when `lookupFile` evaluated a known `from → to` pair. P1-2 does NOT produce these — intent lacks planned-edge information; entries with these statuses appear starting P1-3 / P2 when real edges are observable.
- `boundaryStatus: 'NOT_DECLARED'` → populated when a `from → to` pair was evaluated against triage rules and no rule matched. P1-2 does NOT produce these for the same reason.
- `boundaryStatus: 'NOT_EVALUATED'` → populated when the intent names a file but no planned `from → to` can be derived. OPTIONAL in P1-2 — only emit if useful for debugging. Default: omit from `boundaryChecks[]` entirely, let the empty-array default stand.

Entry shape when emitted:

```ts
{
  intentFile: string,
  submodule: string,
  boundaryStatus: 'ALLOWED' | 'FORBIDDEN' | 'NOT_DECLARED' | 'NOT_EVALUATED',
  rule: ... | null,        // populated ONLY for ALLOWED / FORBIDDEN
  citation: string,
}
```

Pinning in P1-2:

- File intent with no planned edge → `boundary.status = 'NOT_EVALUATED'` in the per-lookup result. `boundaryChecks[]` may omit the entry. Render NEVER says `ALLOWED` / `FORBIDDEN` for these.
- `triage.json` absent → same as above. Never a crash.

Empty-array default preserved from P1-1; P1-3 and P2 may extend `boundaryChecks` when real edges are observable.

### 4.6 Render output additions

Two new sections after the existing three (P1-1) and before `### Planned type escapes`:

- **`### New code candidates`** — `NEW_FILE` / `FILE_STATUS_UNKNOWN` / `NEW_PACKAGE` rows. Each row cites the evidence:
  - `NEW_FILE` → `[grounded, topology.json.nodes does not contain '<file>'; topology.meta.complete = true]`
  - `FILE_STATUS_UNKNOWN` → `[확인 불가, reason: topology incomplete or absent; file existence cannot be grounded]`
  - `NEW_PACKAGE` → `[grounded, package.json.{dependencies, devDependencies, peerDependencies} does not contain '<pkg>']`
  - Boundary rule, when present (ALLOWED / FORBIDDEN), appears as a row sub-line — NEVER as a blocker. When status is `NOT_EVALUATED`, the row states `boundary: not evaluated (no planned from→to edge in intent)` with a `[확인 불가]` citation.

  Section title is deliberately neutral — `New code candidates`, not `okay to create`. A `FORBIDDEN` boundary on a `NEW_FILE` is a legitimate warning, not a contradiction with the section title. Per `canonical/pre-write-gate.md` §6 the gate never blocks; wording follows.

- **`### Watch-for`** — shape lookups (`[확인 불가]`) + `FILE_EXISTS` with high inbound fan-in hub signal. Hub warnings MUST carry the actual count and threshold:
  - `[grounded, topology.json.nodes['<file>'].inboundFanIn = 14, threshold = 10]`
  - Threshold constant: `HUB_INBOUND_FAN_IN_THRESHOLD = 10` (hardcoded in `_lib/pre-write-render.mjs`; config flag is P2).
  - `DEPENDENCY_AVAILABLE` rows do NOT land in Watch-for on sample-count alone. Only when `existingImports.countConfidence === 'grounded'` AND `observedImportCount >= HUB_DEPENDENCY_CONSUMER_THRESHOLD` (also hardcoded) does the dependency qualify for the "deeply entangled" hub signal. Sample-only counts can never trigger Watch-for for dependencies.

Both sections are omitted when empty.

### 5.1 File lookup

### 5.6 Dogfood

Run `pre-write.mjs` on this skill's own repo with a hand-written intent:

```json
{
  "names": ["formatDate"],
  "shapes": [{ "fields": ["year", "month", "day"] }],
  "files": ["_lib/new-helper.mjs"],
  "dependencies": ["dayjs"],
  "plannedTypeEscapes": []
```

## 8. Known risks

- **`topology.json` schema drift.** If `measure-topology.mjs` changes output shape between now and P1-2 landing, file lookup may mis-read. - **`topology.meta.complete` producer contract.** `NEW_FILE` claim requires this field to be explicitly `true`. If `measure-topology.mjs` doesn't yet emit it, P1-2 is limited to `FILE_STATUS_UNKNOWN` for all absent-from-nodes cases — which is correct-by-design, not a bug. Add `topology.meta.complete: true` emission to the topology producer as a P1-2 sub-step OR accept FILE_STATUS_UNKNOWN permanently (both are honest).

P1-2 does not evaluate boundary verdicts (no planned edges), so this risk is dormant until P1-3. When P1-3 activates: pick the most specific (longest-prefix) match.

- **Watch-for threshold noise.** `HUB_INBOUND_FAN_IN_THRESHOLD = 10` is hardcoded. A repo with many hub-sized files produces Watch-for spam. P1-2 accepts the spam — the count+threshold citation makes it verifiable; config flag is P2.

- **Sample-vs-count confusion in dependency lookup.** Easy to accidentally compare `examples.length` to a threshold when the real count lives in `observedImportCount`. Pinning test proves `countConfidence: 'sample-only'` can NEVER trigger Watch-for. Reviewer to verify the render path consults `observedImportCount` + `countConfidence`, never `examples.length`, for Watch-for eligibility.
