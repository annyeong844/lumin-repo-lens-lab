## 4. Output shapes (P1-1)

### 4.1 `_lib/mode-dispatch.mjs`

```ts
dispatchMode(userText: string, cwdMeta: { hasPackageJson: boolean, hasTsconfig: boolean, hasSrcTree: boolean }) → {
  mode: 'pre-write' | 'none',
  rationale: string,
  matchedVerbs: string[],
  matchedGuards: string[],
  compoundGuardPlusVerb: boolean,
  nonTriggerReason?: 'guard-only'
                   | 'no-repo-context'
                   | 'prose-rewrite'
                   | 'comment-typo-fix'
                   | 'pure-inspection',
}
```

`mode: 'none'` when any of the three dispatch conditions fails (guard-only, no repo context, pure inspection). When mode is `'none'`, `nonTriggerReason` classifies WHY. Rationale cites the rule from `mode-contract.md` §2.1 / §3.5.

**Relationship with `pre-write.mjs` CLI (Option A — CLI is explicit):** `dispatchMode` is a pure library function for future SKILL-runtime / orchestrator use. The `pre-write.mjs` CLI does NOT call `dispatchMode` — invoking the CLI IS the explicit dispatch decision, made by the caller. CLI assumes mode `'pre-write'` and always runs the lookup / render / artifact pipeline. `dispatchMode` ships in P1-1 as a library because `test-mode-dispatch.mjs` pins the rule table, but CLI wiring remains a separate concern. This avoids the round-trip "CLI calls dispatchMode, dispatchMode says 'none', CLI no-ops" which would make the mode-contract §2.2 non-triggers harder to reason about.

### 4.2 `_lib/pre-write-intent.mjs`

```ts
validateIntent(raw: unknown) → {
  ok: true,
  intent: {

    shapes: { fields: string[] }[],

}

type PlannedEscape = {

```

### 4.3 `_lib/pre-write-lookup-name.mjs`

```ts
lookupName(intentName, { symbols, canonicalClaims, unresolvedInternalSpecifiers, filesWithParseErrors }) → {
  intentName,
  result: 'EXISTS'
        | 'EXISTS_MULTIPLE'
        | 'CANONICAL_EXISTS_AND_EXISTS'
        | 'CANONICAL_EXISTS_AST_ABSENT'
        | 'CANONICAL_EXISTS_AST_DISAGREE'
        | 'NOT_OBSERVED',

  // identities[] is always an array, possibly empty (NOT_OBSERVED) or
  // possibly length > 1 (EXISTS_MULTIPLE / AST observes two or more
  // identities sharing the same exportedName). `identity-and-alias.md`
  // §3 forbids name-keyed collapsing — each owner file gets its own row.
  identities: Array<{
    identity: string,                  // 'ownerFile::exportedName'
    ownerFile: string,
    exportedName: string,
    fanIn: number | null,
    fanInConfidence: 'grounded' | 'unavailable',
    anyContamination: {
      state: 'clean'
           | 'unknown-surface-only'
           | 'has-any-only'
           | 'any-contaminated'
           | 'severely-any-contaminated'
           | 'capability-absent',
      labels?: string[],
      measurements?: object,
    },
    resolverConfidence: 'high' | 'medium' | 'low',
  }>,

  // Canonical owner claim when recognized-schema canonical file exists
  // and contains a row for intentName. Single object (not an array) —
  // canonical-draft output asserts one owner per name in Single-owner
  // tables by construction. `canonicalAstStatus` is the RAW observation
  // (aligned / ast-absent / owner-disagrees). The formal "CANONICAL
  // DRIFT:" rendered warning is deferred to P1-3; P1-1 only preserves
  // the state so P1-3 can emit.
  canonicalClaim: {
    ownerFile: string,
    line: number,
    file: string,
    section: string,
  } | null,
  canonicalAstStatus: 'not-consulted'   // canonical absent OR schema unrecognized
                    | 'aligned'          // canonical claim matches at least one AST identity
                    | 'ast-absent'       // canonical says X, AST has zero identities
                    | 'owner-disagrees', // canonical says owner X, AST observes owner Y instead

  nearNames: {                          // search hints only, NEVER a reuse claim
    name: string,
    ownerFile: string,
    distance: number,
  }[],

  citations: string[],                  // every claim-bearing row in this result
}
```

Shape notes:
- `result === 'EXISTS_MULTIPLE'` fires when `identities.length ≥ 2` with no canonical disambiguation. Render all identities side-by-side; never silently pick one.
- When canonical exists but AST has MORE identities than canonical lists (e.g. canonical says owner X, AST has X + Y), `result === 'CANONICAL_EXISTS_AND_EXISTS'` with `identities.length ≥ 2` AND `canonicalAstStatus: 'aligned'` (X matches) — but the render shows the extra Y as well, so Claude sees what canonical did not declare. This is NOT a drift per se; it is a canon-draft staleness hint, and P1-3's drift logic subsumes it later.
- `canonicalAstStatus: 'owner-disagrees'` replaces the v1 `CANONICAL_AST_DRIFT` result. State is preserved; the rendered `CANONICAL DRIFT:` warning is a P1-3 concern per §5.9 of `docs/history/phases/p1/session.md`. P1-1 render surfaces the disagreement with a neutral citation (see §4.4).

### 5.1 Mode dispatch

**Test first:** `tests/test-mode-dispatch.mjs`. Cases:

- Guard alone (`"찾아줘"`) → `mode: 'none'`, `nonTriggerReason: 'guard-only'`.
- Guard alone English (`"explain this"`) → `mode: 'none'`, `nonTriggerReason: 'guard-only'`.
- Write verb alone (`"만들어줘"`) → `mode: 'pre-write'`.
- Compound guard + verb (`"기존 helper 찾아서 연결해줘"`) → `mode: 'pre-write'`, `compoundGuardPlusVerb: true`.
- No repo context (`hasPackageJson: false`, `hasTsconfig: false`, `hasSrcTree: false`) → `mode: 'none'`, `nonTriggerReason: 'no-repo-context'`.
- **Prose rewrite (`"README 다듬어줘"`) → `mode: 'none'`, `nonTriggerReason: 'prose-rewrite'`.** Per `canonical/mode-contract.md` §2.2 (prose rewrite is a listed non-trigger), pre-write MUST NOT fire here. The dispatcher detects prose-only requests via a small heuristic file-name hint inside `userText` — explicit mentions of `README` / `CHANGELOG` / `*.md` / `docs/` / `"prose"` / `"다듬" + doc-file` — and returns `'none'` with the prose-rewrite reason. The "fire + no-op in calling code" approach is FORBIDDEN because it contradicts canonical and makes the non-trigger semantics fragile.
- Comment typo fix (`"주석 오타 고쳐줘"`, `"fix comment typo"`) → `mode: 'none'`, `nonTriggerReason: 'comment-typo-fix'`. Detected via explicit `"주석"` / `"comment"` + `"오타"` / `"typo"` combination. More conservative than the general "fix" verb alone.
- Pure inspection (`"이 코드 어떻게 동작해요?"`) → `mode: 'none'`, `nonTriggerReason: 'pure-inspection'`.

**Impl:** `_lib/mode-dispatch.mjs`. Pure function; reads only `userText` + `cwdMeta`. Verb + guard lists imported from constants at the top of the file (matching canonical/mode-contract.md §3.1 / §3.2 / §3.4). Compound detection: if at least one verb AND at least one guard both match, `compoundGuardPlusVerb = true`, mode = `pre-write`. Non-trigger detection precedence: `no-repo-context` (highest) → `prose-rewrite` → `comment-typo-fix` → `pure-inspection` (lowest). First match sets `nonTriggerReason`.

**Exit:** test green; every §2.2 non-trigger from mode-contract has a case that returns `mode: 'none'` with a correct `nonTriggerReason`; no case requires calling-code to "no-op after dispatch fires".

### 5.3 Name lookup

**Test first:** `tests/test-pre-write-lookup-name.mjs`. Cases (fixtures built with in-memory `symbols` objects + synthetic `canonicalClaims`):

Structural (independent of contamination):
- `CANONICAL_EXISTS_AND_EXISTS` — canonical says owner X, AST agrees on owner X. `identities.length === 1`, `canonicalAstStatus: 'aligned'`.
- `CANONICAL_EXISTS_AST_ABSENT` — canonical says owner X, AST has no definition. `identities.length === 0`, `canonicalAstStatus: 'ast-absent'`.
- **`CANONICAL_EXISTS_AST_DISAGREE`** — canonical says X, AST observes Y (and NOT X). `identities.length ≥ 1`, `canonicalAstStatus: 'owner-disagrees'`. Result PRESERVES the disagreement state; result MUST NOT emit a rendered `CANONICAL DRIFT:` warning (that is P1-3's §5.9 concern). P1-1 render surfaces both sides with a neutral citation instead — see §4.4.
- Canonical schema unrecognized (empty `canonicalClaims`) → falls through to AST-only; no CANONICAL_EXISTS variant fires. `canonicalAstStatus: 'not-consulted'`.
- `EXISTS` — AST only; no canonical claim. `identities.length === 1`.
- **`EXISTS_MULTIPLE`** — AST has 2+ identities sharing the `exportedName`. Result carries the full `identities[]` array (never picks one); render shows each owner side-by-side with per-identity fan-in and contamination. Pinning: `identities.length ≥ 2`; `identity-and-alias.md` §3 forbids name-keyed collapsing.
- `NOT_OBSERVED` with near-name hints (up to 5, edit-distance ≤ 2 or shared prefix ≥ 4). `identities.length === 0`.
- `NOT_OBSERVED` with zero near-name hints. `identities.length === 0`, `nearNames: []`.

Multiple-identity edge cases (all pinned):
- Canonical declares owner X for `User`; AST has X AND Y. Result: `CANONICAL_EXISTS_AND_EXISTS` (canonical is aligned with X) with `identities.length === 2` — P1-1 surfaces the extra Y so Claude sees what canon-draft did not declare. This is NOT drift; the P1-3 drift logic subsumes the "canon missed Y" case if ever promoted.
- Canonical declares owner X for `User`; AST has Y only (not X). Result: `CANONICAL_EXISTS_AST_DISAGREE` with `identities.length === 1`.
- Canonical declares owner X; AST has both X and Y; AND canonical schema unrecognized → canonical not consulted, result: `EXISTS_MULTIPLE`, `canonicalAstStatus: 'not-consulted'`.

Fan-in:
- Identity-keyed fan-in present → grounded number.
- `symbols.meta.supports.identityFanIn === false` OR `fanInByIdentity` absent → `fanIn: null`, `fanInConfidence: 'unavailable'`, `[확인 불가]` citation.
- **`symbols.topSymbolFanIn[name]` is NEVER read** — pinning test: even when `topSymbolFanIn` contains a value for the name, `lookupName` ignores it.

Contamination (with `supports.anyContamination: true`):
- `anyContamination` absent → `state: 'clean'`.
- `labels: ['unknown-surface']` only → `state: 'unknown-surface-only'`; rendered as `[grounded structural, semantic caution: unknown-surface]`. **Never "contaminated".**
- `labels: ['has-any']` only → `state: 'has-any-only'`; rendered as `[grounded structural, any signal present, semantic caution]`.
- `labels: ['has-any', 'any-contaminated']` → `state: 'any-contaminated'`; `[degraded, any-contaminated, confidence: low]`.
- `labels: ['has-any', 'any-contaminated', 'severely-any-contaminated']` → `state: 'severely-any-contaminated'`; same degraded citation with severe measurements.

Contamination (with `supports.anyContamination: false`):
- Absent annotation → `state: 'capability-absent'`; `[확인 불가, reason: producer did not emit anyContamination capability]`. **NOT clean.**

Resolver confidence:
- No shape match against unresolved specifiers → confidence unchanged.
- Path-shape match via `specifierCouldMatchFile` → demote one level (high → medium → low → `[확인 불가]`).
- File in `filesWithParseErrors` → resolver confidence drops AND a parse-status taint is emitted.

**Impl:** `_lib/pre-write-lookup-name.mjs`. Pure function; no I/O. Consumes injected `symbols` / `canonicalClaims`. Implements the §5.3 algorithm from `docs/history/phases/p1/session.md` 1:1.

Near-name algorithm (O(N) in practice thanks to cheap filters first):

1. Iterate every key `ownerFile::exportedName` in `symbols.defIndex` → candidate name.
2. **Cheap filter A (length delta).** Skip if `|candidate.length - intentName.length| > 2`. Levenshtein distance ≥ 3 is guaranteed when length delta ≥ 3; no need to compute.
3. **Cheap filter B (shared prefix).** If `sharedPrefix(candidate, intentName) ≥ 4`, mark candidate as a prefix-match hit (distance computed lazily below). Otherwise still eligible via distance filter.
4. **Expensive filter (Levenshtein ≤ 2).** Computed only on candidates that passed (2); if distance ≤ 2 OR prefix-match flag is set, candidate qualifies.
5. Sort qualifying candidates: distance ascending, then name alphabetical.
6. Cap to top 5 results.
7. Each near-name row is tagged `isReuseCandidate: false` in the result model; render renders under `### Search hints (not reuse candidates)` with the "search hint only, NOT a grounded reuse claim" citation text.

If near-name scanning is ever observed to be slow on very large repos (> 10k identities), P1-3 can add a `--no-near-names` flag; P1-1 defaults near-names on. The cheap filters keep the common case O(N) with tiny constants, not O(N × M × |levenshtein|).

**Exit:** every branch covered; pinning test for `topSymbolFanIn` non-use; clean / unknown / mild / any / severe / capability-absent each distinct rendering.

## 8. Known risks

- **Near-name cost on large repos.** Edit-distance over all `defIndex` names is O(N × M). Cap at 5 results and short-circuit once both filters fire; if perf becomes an issue on >10k-symbol repos, add a `--no-near-names` flag in P1-3. Not a P1-1 blocker.
- **Canonical parsing on session start.** Parser is fast (≤ O(lines-in-canon-file)), but if canonical/ grows large, consider caching. P1-1 does not cache — every CLI invocation re-parses. Revisit if dogfood latency is concerning.
- **Intent hash stability.** `JSON.stringify` with key sorting is the normalization; if the intent ever contains numeric values with representations that differ across engines (rare), the hash would drift. P1-1 pins normalization to `JSON.stringify(intent, Object.keys(intent).sort())` recursively; a helper function does the deep sort. Test `test-pre-write-advisory-artifact.mjs` includes an intent-normalization stability check.
- **CLI stdin + Windows.** `--intent -` read via `process.stdin` works on Windows but the test harness should use `--intent <file>` for fixture simplicity. stdin variant gets a manual smoke test at §5.6 exit; no CI pinning.
