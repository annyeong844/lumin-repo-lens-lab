# docs/history/phases/p6/p6-0.md - Measurement Harness And Gates

> **Phase:** P6-0 - measurement foundation for TypeScript precision and
> throughput work. This phase must land before any P6-1 precision changes.
> **Role:** define and implement one repeatable baseline artifact that records
> corpus identity, candidate counts, adjudication protocol, runtime metrics,
> P3/P5 schema round-trip health, and readiness gates.
> **Status:** session plan, v2 (2026-04-22 reviewer P0/P1 absorbed:
> measurement-first order, FP denominator, sampling protocol,
> `not_applicable` policy, immutable corpus registry, missing-artifact
> unavailable/null semantics, explicit `--adjudication` input, schema
> round-trip Green blocker, dirty-worktree snapshot rule, standalone-v1
> boundary, machine-readable readiness reasons).
> **Last updated:** 2026-04-22

---

## 0. Implementation Rule

Do not start P6-1 Package/Public Surface Model, P6-2 Symbol-Level Public API,
or any precision implementation until P6-0 produces a baseline
`p6-measurement.json` for at least one pinned corpus entry.

P6-0 is allowed to add measurement code and tests. It is not allowed to change
dead-export classification logic, resolver semantics, public API policy,
namespace/dynamic import precision, generated-file policy, checker behavior,
or AST cache behavior.

## 2. Scope

### 2.1 In

- One measurement artifact: `<output>/p6-measurement.json`.
- Measurement-only merge mode for combining per-corpus
  `p6-measurement.json` artifacts into one readiness artifact.
- Corpus registry shape and immutable snapshot validation.
- Candidate-count extraction from existing artifacts.
- Sampling/adjudication schema. P6-0 does not adjudicate automatically; it
  records the protocol and accepts adjudication entries.
- Readiness-gate computation from adjudication entries.
- Runtime summary derived from `manifest.commandsRun` plus P6-0 wall time.
- P3/P5 schema round-trip status fields.
- Tests for denominator behavior, corpus immutability, and gate computation.

### 2.2 Out

- Any precision improvement.
- Any automatic candidate adjudication.
- Any automatic deletion or code modification in a benchmark repo.
- Marketplace publication.
- Shared AST cache implementation.
- TypeScript checker integration.
- Symbol-level public API implementation.

## 3. Artifact

P6-0 writes one JSON artifact:

```text
<output>/p6-measurement.json
```

The first implementation should keep this as one file. Splitting into
`p6-corpus.json`, `p6-runtime.json`, or `p6-fp-adjudication.json` is deferred
until the single artifact becomes too large to review.

P6-0b may merge multiple already-produced measurement artifacts without
rerunning analyzers or changing precision behavior:

```bash
node p6-measurement.mjs --root <skill-repo> --output <dir> \
  --merge <repo-a>/p6-measurement.json \
  --merge <repo-b>/p6-measurement.json
```

Merge mode concatenates corpus/adjudication entries, sums candidate/runtime
counts when every input measured the field, prefixes per-source schema/canon
signals with the corpus name, and recomputes readiness. It must not adjudicate
or mutate benchmark repositories.

### 3.1 Top-Level Shape

```json
{
  "schemaVersion": "p6-measurement.v1",
  "meta": {},
  "corpus": [],
  "candidateCounts": {},
  "adjudication": {},
  "runtime": {},
  "schemaRoundTrip": {},
  "readiness": {}
}
```

## 4. Corpus Registry

Each corpus entry must identify an immutable benchmark snapshot.

```json
{
  "name": "duyet-monorepo",
  "repo": "local-or-remote-id",
  "commit": "abc123",
  "snapshotId": null,
  "worktreeDirty": false,
  "contentHash": null,
  "locBucket": "50k",
  "packageManager": "pnpm",
  "includeTests": false,
  "exclude": ["node_modules", "dist"],
  "reason": "multi-app workspace with scoped tsconfig aliases"
}
```

Rules:

- `name` is stable across runs.
- `repo` may be a remote URL, local path label, or benchmark registry id.
- `commit` is preferred. If the corpus is not a git checkout, `snapshotId`
  must contain an equivalent immutable id.
- Green readiness is forbidden when any corpus entry lacks both `commit` and
  `snapshotId`.
- If `worktreeDirty === true`, Green readiness is forbidden unless
  `snapshotId` or `contentHash` captures the exact dirty state.
- `locBucket` is one of `25k`, `50k`, `100k`, or `other`.
- `includeTests` and `exclude` must mirror the actual scan range.

## 5. Candidate Population Mapping

### 5.1 Primary FP Population

Review-visible cleanup candidates are:

```text
fix-plan.safeFixes + fix-plan.reviewFixes
```

These are the primary FP denominator because they are the candidates a user
would see as removable, demotable, or cleanup-actionable.

### 5.2 Separate Populations

- `SAFE_FIX` precision is computed separately from `fix-plan.safeFixes`.
- A measured zero `SAFE_FIX` population is not a failure and must not be
  reported as an unknown FP rate. It means autonomous cleanup precision is
  not measured; Green readiness is blocked with reason code
  `safe-fix-population-empty`.
- Raw Tier C is reported from `dead-classify.json` bucket C, but is not a
  claim and is not the marketplace cleanup denominator.
- `DEGRADED` is reported separately. It is excluded from the primary cleanup
  denominator unless a future UI explicitly shows it as cleanup-actionable.
- `MUTED` is reported separately as policy behavior, not as a cleanup
  candidate.
- Canon drift candidates are register/schema drift signals, not dead-export
  cleanup candidates.

### 5.3 Candidate Counts Shape

```json
{
  "candidateCounts": {
    "available": true,
    "missingArtifacts": [],
    "reviewVisibleCleanup": 0,
    "safeFix": 0,
    "reviewFix": 0,
    "degraded": 0,
    "muted": 0,
    "rawTierC": 0,
    "canonDrift": {
      "available": true,
      "missingArtifacts": [],
      "total": 0,
      "perSource": {}
    }
  }
}
```

Rules:

- `0` means "measured and zero".
- `null` means "not measured".
- If `fix-plan.json` is absent, set `candidateCounts.available = false`,
  add `fix-plan.json` to `candidateCounts.missingArtifacts`, set cleanup
  counts to `null`, force readiness `Red`, and include reason code
  `candidate-counts-unavailable`.
- If `dead-classify.json` is absent, set `rawTierC = null` and add the
  artifact name to `missingArtifacts`; do not silently report raw Tier C as
  zero.
- If `canon-drift.json` is absent, set
  `candidateCounts.canonDrift.available = false`, set
  `canonDrift.total = null`, and add the artifact name to
  `canonDrift.missingArtifacts`.

## 6. Adjudication Protocol

P6-0 v1 accepts adjudication through an explicit JSON file:

```bash
node p6-measurement.mjs \
  --root <repo> \
  --output <dir> \
  --corpus-name <name> \
  --repo <id> \
  --commit <sha> \
  --snapshot-id <id> \
  --content-hash <hash> \
  --worktree-dirty true|false \
  --adjudication <adjudication.json>
```

If `--adjudication` is omitted, `adjudication.entries = []`, readiness is
`Red`, and readiness reasons include `fp-rate-unknown`.

`--snapshot-id`, `--content-hash`, and `--worktree-dirty` are optional.
If omitted, the CLI may auto-detect git commit and dirty state when possible.
If dirty state cannot be determined, Green readiness is forbidden.

### 6.1 Verdicts

Allowed verdicts:

- `true_dead`
- `false_positive`
- `inconclusive`
- `not_applicable`

FP rate formula:

```text
false_positives / (true_dead + false_positives)
```

`inconclusive` and `not_applicable` are excluded from numerator and
denominator. Both are still reported as sampling-quality signals.

Examples of `not_applicable`:

- candidate is outside cleanup domain;
- generated-policy-only;
- canon-drift-only;
- fixture malformed;
- benchmark artifact missing required context.

### 6.2 Sampling Rules

For each corpus entry:

- Adjudicate at least 50 candidates, or all candidates if fewer than 50.
- Stratify by `SAFE_FIX`, `REVIEW_FIX`, `DEGRADED`, and `MUTED`.
- Keep `DEGRADED` and `MUTED` samples visible even though they do not enter
  the primary cleanup denominator by default.
- Record FP ledger references when the reviewer identifies a known pattern.

### 6.3 Adjudication Item Shape

```json
{
  "id": "duyet-monorepo:SAFE_FIX:0001",
  "corpusName": "duyet-monorepo",
  "sourceArtifact": "fix-plan.json",
  "tier": "SAFE_FIX",
  "bucket": "C",
  "file": "packages/example/src/foo.ts",
  "symbol": "Foo",
  "line": 12,
  "action": "remove-symbol",
  "verdict": "false_positive",
  "reason": "consumed by unresolved scoped alias",
  "fpLedgerRefs": ["FP-36"],
  "reviewer": "manual",
  "notes": ""
}
```

## 7. Runtime Metrics

P6-0 must preserve existing `manifest.commandsRun` timing and add a P6 summary.

```json
{
  "runtime": {
    "wallMs": 0,
    "childProcessCount": 0,
    "steps": [
      {
        "step": "build-symbol-graph.mjs",
        "status": "ok",
        "ms": 0
      }
    ],
    "parseCount": null,
    "fileWalkMs": null,
    "resolverConstructionMs": null,
    "cacheHits": null,
    "cacheMisses": null
  }
}
```

Rules:

- `parseCount`, `fileWalkMs`, `resolverConstructionMs`, `cacheHits`, and
  `cacheMisses` may be `null` in P6-0 if producers do not expose them yet.
- P6-6 must replace the relevant `null` fields with measured values.
- P6-6 throughput changes must preserve candidate counts unless a precision
  phase explicitly changed the expected counts.

## 8. Schema Round-Trip Health

P6-0 records whether P3/P5 can round-trip the promoted canonical shape.
P6-0 v1 is record-based: it reads existing `manifest.checkCanon` and/or
`canon-drift.json`. It does not run P3/P5 itself. If neither artifact is
present, `schemaRoundTrip.attempted = false` and Green is blocked.

```json
{
  "schemaRoundTrip": {
    "attempted": false,
    "sources": {
      "type-ownership": { "status": "not-run", "driftCount": null },
      "helper-registry": { "status": "not-run", "driftCount": null },
      "topology": { "status": "not-run", "driftCount": null },
      "naming": { "status": "not-run", "driftCount": null }
    },
    "knownSchemaDriftBugs": []
  }
}
```

Green readiness requires `knownSchemaDriftBugs.length === 0`.
Green readiness also requires `schemaRoundTrip.attempted === true`. If
`attempted === false`, readiness cannot be Green and reasons must include
`schema-roundtrip-not-attempted`.

## 9. Readiness Computation

Readiness gate values:

- `Red`
- `Yellow`
- `Green`

### 9.1 Red

Any of:

- FP rate unknown;
- review-visible cleanup FP > 25%;
- `SAFE_FIX` FP >= 5%;
- candidate counts unavailable;
- schema round-trip not attempted;
- any corpus entry lacks immutable identity;
- any corpus entry has `worktreeDirty === true` without `snapshotId` or
  `contentHash`;
- any corpus entry has unknown dirty state;
- unresolved HIGH finding blocks measurement trust.

### 9.2 Yellow

Any of these non-Red states:

- `SAFE_FIX` FP < 5% when the `SAFE_FIX` population is non-empty, but
  another Green requirement is incomplete;
- `SAFE_FIX` population is measured zero, which blocks Green/autonomous
  cleanup claims without making the run Red;
- review-visible cleanup FP is between 10% and 25%, or benchmark is
  incomplete;
- no known P3/P5 schema drift bug.

### 9.3 Green

All of:

- non-empty `SAFE_FIX` population with measured `SAFE_FIX` FP < 5%;
- review-visible cleanup FP < 10%;
- at least two non-trivial TS corpus entries;
- at least 50 adjudicated candidates per corpus entry, or all candidates if
  fewer than 50;
- every corpus entry has `commit` or `snapshotId`;
- every dirty corpus entry has `snapshotId` or `contentHash`;
- schema round-trip attempted;
- zero known P3/P5 schema drift bugs;
- no unresolved HIGH findings in the FP ledger.

### 9.4 Readiness Shape

```json
{
  "readiness": {
    "gate": "Red",
    "reasons": [
      {
        "code": "fp-rate-unknown",
        "severity": "red",
        "detail": "no adjudication entries supplied"
      }
    ],
    "safeFix": {
      "falsePositives": 0,
      "trueDead": 0,
      "inconclusive": 0,
      "notApplicable": 0,
      "fpRate": null
    },
    "reviewVisibleCleanup": {
      "falsePositives": 0,
      "trueDead": 0,
      "inconclusive": 0,
      "notApplicable": 0,
      "fpRate": null
    }
  }
}
```

Reason codes are machine-readable. Detail text may vary, but `code` and
`severity` must remain stable enough for tests and future CI gates.

Initial reason codes:

- `candidate-counts-unavailable`
- `fp-rate-unknown`
- `safe-fix-fp-threshold`
- `review-visible-fp-threshold`
- `corpus-identity-missing`
- `dirty-worktree-without-snapshot`
- `dirty-worktree-unknown`
- `schema-roundtrip-not-attempted`
- `schema-drift-known`
- `unresolved-high-finding`
- `safe-fix-population-empty`
- `benchmark-incomplete`

## 10. Tests

P6-0 should add focused tests before implementation.

Recommended test surface:

- `tests/test-p6-measurement.mjs` NEW.

Minimum assertions:

- `not_applicable` excluded from FP numerator and denominator.
- `inconclusive` excluded from FP numerator and denominator.
- missing `fix-plan.json` yields unavailable/null counts, not zero counts.
- missing adjudication file yields Red with `fp-rate-unknown`.
- `SAFE_FIX` FP >= 5% forces Red.
- review-visible cleanup FP > 25% forces Red.
- schema round-trip not attempted blocks Green.
- dirty worktree without `snapshotId` or `contentHash` blocks Green.
- unknown dirty state blocks Green.
- measured-zero `SAFE_FIX` population yields Yellow with
  `safe-fix-population-empty`, not Red with `fp-rate-unknown`.
- review-visible cleanup FP < 10% and `SAFE_FIX` FP < 5% can reach Green
  only when corpus requirements and schema round-trip requirements are met.
- missing corpus `commit` and `snapshotId` blocks Green.
- review-visible cleanup count maps to `safeFixes + reviewFixes`.
- `DEGRADED` and `MUTED` are reported but excluded from the primary cleanup
  denominator.

P6-0 v1 is standalone only:

- No `audit-repo.mjs` integration.
- No default profile integration.
- No normal-audit overhead.

Deferred design questions:

- Whether P3/P5 schema round-trip execution is added in P6-0b or P6-final.

The first implementation remains standalone to avoid adding P6 measurement
overhead to normal audits.

## 12. Marketplace Wording Reminder

Until P6-final reruns this harness after P6-1 through P6-6, allowed wording
remains advisory:

- "Grounded structural audit for TS/JS/Python/Go repositories."
- "Produces evidence-backed cleanup candidates with confidence labels."
- "Detects canonical drift across type, helper, topology, and naming
  registers."

Not allowed:

- "Automatically removes dead code safely."
- "Perfect TypeScript semantic analysis."
- "Zero false positives."
- "Marketplace-ready autonomous cleanup."
