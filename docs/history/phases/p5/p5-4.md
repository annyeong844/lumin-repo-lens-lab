**Exit codes (standalone, session.md v2 §5.1 P0-1 + P0-4):**

- **Single-source naming** — identical matrix to helper-registry (fresh AST pass):
  - 0 clean / 1 drift / 2 parse-error / missing canon / CLI fatal.
  - Missing/corrupt symbols.json is irrelevant for naming (not consumed); no warning, no exit-2 escalation.
- **`--source all`** — aggregate by *checked-source* set, NOT by max-of-per-source-exits (reviewer P0-2, 2026-04-22 v1→v2). A missing canon alone is NOT a hard failure in all-mode; it records a skip and the other sources can still report clean. Reference algorithm:

```js
function aggregateAllSourceExit(perSource) {
  const entries = Object.values(perSource);
  const skipped = entries.filter((e) => e.status === 'skipped-missing-canon');
  const failed  = entries.filter((e) =>
    e.status === 'parse-error' || e.status === 'skipped-unrecognized-schema');
  const checked = entries.filter((e) =>
    e.status === 'clean' || e.status === 'drift');

  // 1. If no source could even be checked (all skipped + failed), exit 2.
  if (checked.length === 0) return 2;
  // 2. Any failed source is a hard error regardless of other sources.
  if (failed.length > 0) return 2;
  // 3. Any drift among checked sources escalates to 1.
  if (checked.some((e) => e.status === 'drift')) return 1;
  // 4. All checked sources clean.
  return 0;
}
```

Rule table (checked = status ∈ {clean, drift}; failed = {parse-error, skipped-unrecognized-schema}; skipped = {skipped-missing-canon}):

| Aggregate state | Exit |
|---|---:|
| All 4 skipped-missing-canon | 2 |
| Any failed (parse-error / unrecognized-schema) | 2 |
| checked.length === 0 (all skipped or failed) | 2 |
| Any drift among checked sources | 1 |
| All checked sources clean (others skipped ok) | 0 |

Concrete worked examples:

| type-ownership | helper-registry | topology | naming | Exit |
|---|---|---|---|---:|
| clean | skipped | skipped | skipped | 0 |
| drift | skipped | skipped | skipped | 1 |
| clean | drift | skipped | clean | 1 |
| parse-error | clean | clean | clean | 2 |
| skipped | skipped | skipped | skipped | 2 |

### 4.4 Orchestrator — `audit-repo.mjs --check-canon`

```
audit-repo.mjs --check-canon [--sources <csv>] [--strict-check-canon]
```

- `--sources <csv>` — optional comma-separated subset of `type-ownership,helper-registry,topology,naming,all`. Default: all 4 named sources. `--sources all` expands to the 4 named sources (NOT the CLI's `--source all` — orchestrator iterates them individually to get granular `perSource` data).
- `--strict-check-canon` — boolean flag, default false.

**Child-process spawn contract (reviewer P0-3, 2026-04-22 v1→v2):** `execFileSync` throws on any non-zero exit. Both drift (exit 1) and parse-error/missing-canon (exit 2) are LEGITIMATE per-source outcomes that must be RECORDED into the manifest, not treated as spawn failures. The orchestrator MUST wrap every spawn in try/catch and interpret `err.status` (or `err.code === 'ENOENT'` for true spawn failures).

```js
// Orchestrator per-source loop sketch:
let childResult;
try {
  execFileSync(process.execPath, [checkCanonCli, '--source', name, '--root', ROOT, '--output', OUT],
    { stdio: ['ignore', 'pipe', 'pipe'] });
  childResult = { ran: true, exitCode: 0 };
} catch (err) {
  const exitCode = typeof err.status === 'number' ? err.status : 1;
  // exit 1 = drift (ran successfully); exit 2 = missing-canon or parse-error (attempted);
  // any other = true spawn failure (err.code / err.signal).
  const isRealSpawnFailure = err.status === null || err.status === undefined;
  childResult = {
    ran: !isRealSpawnFailure,    // child produced a structured result (even if exit 2)
    exitCode,
  };
}
// Then read <output>/canon-drift.json for the per-source status + driftCount.
```

**Shape definitions (pinned):**
- `ran` (top-level) — at least one requested source produced a structured `perSource` result (drift, clean, or attempted-but-failed). `false` only when unknown `--sources` gate rejects the invocation OR every spawn hit a true spawn failure (ENOENT etc).
- `strict` — mirrors whether `--strict-check-canon` was set (for downstream consumers to understand why an exit code came out a certain way).
- `summary.sourcesChecked` — count of `perSource[source].status ∈ {clean, drift}`.
- `summary.sourcesSkipped` — count of `perSource[source].status === 'skipped-missing-canon'`.
- `summary.sourcesFailed` — count of `perSource[source].status ∈ {parse-error, skipped-unrecognized-schema}`.
- `summary.driftCount` — sum of `driftCounts` values.
- `perSource[s].ran` — `true` iff the child process produced a structured canon-drift.json entry for `s`. Exit 1 (drift) and exit 2 (attempted-but-failed) both count as `ran: true`. Only a true ENOENT-style spawn failure is `ran: false`.
- `perSource[s].status` — copied directly from child's `canon-drift.json.perSource[s].status`. The 5-value domain from canon-drift.md §6 applies.
- `perSource[s].exitCode` — the child process's numeric exit code.

**Advisory default (no --strict-check-canon):**
- Orchestrator exit 0 if `manifest.checkCanon.ran === true`.
- Orchestrator exit 1 ONLY if `--sources` contained unknown values (same contract as `--canon-draft` block).
- Child exit 1 (drift) or exit 2 (parse-error / skipped-missing-canon) does NOT promote orchestrator exit code.

**Strict mode (--strict-check-canon):**
- `summary.driftCount > 0` → orchestrator exit 1.
- `summary.sourcesChecked === 0` (all sources either failed or skipped) → orchestrator exit 2.
- Mixed: some drift + some clean → exit 1. Some failed + some clean → exit 0 (checked sources are fine). Some skipped + some clean → exit 0. Strict escalates ONLY on drift in checked sources OR complete checked-source absence.

### 4.5 Category dispatch (naming drift algorithm)

Two sub-diffs:

**Sub-diff 1 — Cohorts:**
- Identity keyspace = canon cohort identity set (fileCohort submodule string OR symbolCohort `submodule::kind`).
- Fresh-only → `cohort-added`.
- Canon-only → `cohort-removed`.
- Both sides: if `canon.dominantConvention !== fresh.dominantConvention` OR `canon.label !== fresh.label` (where label reflects `consistent-<convention>` vs `mixed-convention` vs `insufficient-evidence`) → `cohort-convention-shifted`.
- No other cohort-level drift category (e.g., ConsistencyRate deltas are NOT a v1 category).

**Sub-diff 2 — Outliers:**
- Identity keyspace = per-item outlier identity (file path OR `ownerFile::exportedName`).
- Fresh-only → `new-outlier-introduced`.
- Canon-only → `outlier-resolved`.
- Both sides with differing `ObservedConvention` → NOT a v1 category (no `outlier-convention-shifted` in §3.1); silent.

**Drift record assembly:** `makeDriftRecord({ kind: 'naming-drift', category, identity, canon, fresh })`. Families auto-attached per `CATEGORY_TO_FAMILY` (all 5 already present).

### 4.6 Scan-range flag forwarding per source (reviewer P0-5, 2026-04-22 v1→v2)

`check-canon.mjs` already accepts `--include-tests` / `--no-include-tests` / `--production` / `--exclude <pattern>` via the shared `parseCliArgs` helper. Their behavior VARIES by source because the observation model differs:

| Source | Observation source | Scan-range flags active? | Notes |
|---|---|:-:|---|
| `type-ownership` | `symbols.json` projection | indirect | symbols.json was produced with its own scope; flags influence only the stale-freshness warning (which file paths are checked for mtime staleness). They do NOT re-run symbol extraction. |
| `helper-registry` | fresh AST pass (P5-2) | **yes** | `collectFiles(root, { includeTests, exclude })` determines which files enter the helper collector. Same for naming. |

| `naming` | fresh AST pass (P5-4) | **yes** | Identical to helper-registry — `collectFiles(root, { includeTests, exclude })` is load-bearing. Cohort membership and outlier detection are highly sensitive to scan range; forwarding flags is MANDATORY. |

**Under `--source all`:** per-source observation models are preserved. The CLI forwards scan-range flags uniformly; each engine honors or ignores them based on its model:
- type-ownership + topology: flags affect only the stale-freshness warning surface (not re-run).
- helper-registry + naming: flags active; `collectFiles` is called with them in each engine's branch.

Required CLI tests (§5.1):

- `check-canon --source all --no-include-tests` → naming + helper-registry honor the flag; type-ownership + topology unchanged (their producers decided scope).

## 8. Known risks

- **P5-4-R1: `--source all` performance.** 4 sequential engines, each with AST / topology.json parse. On large repos this is additive — 15-30s total. **Mitigation:** acceptable for dev / CI; parallelize via worker_threads is a v2 option.
- **P5-4-R2: Orchestrator spawn cost.** Each `execFileSync` of `check-canon.mjs` bootstraps a fresh Node process. For 4 sources × ~1s node startup = 4s overhead on top of engine runtime. **Mitigation:** accept v1; consider in-process invocation (import + call) in orchestrator v2 if cost materializes.
- **P5-4-R3: `submoduleOf` derivation location.** If P3-4 keeps `submoduleOf` inline in `generate-canon-draft.mjs`, P5-4 CLI needs a small preparatory refactor to expose it as a leaf export. **Mitigation:** check during Step 0; if inline, extract first as a single-commit preparatory change; fall back to re-implementing the derivation in the CLI (duplicated code, flagged for later cleanup).
- **P5-4-R4: `--source all` exit-code ambiguity under mixed states.** When one checked source drifted (would be exit 1 standalone) and another checked source parse-errored (would be exit 2 standalone), the checked-source aggregation rule (§4.3) returns exit 2 — failed checked sources outrank drift. This is NOT a raw max over all per-source exits: `skipped-missing-canon` sources are excluded from the checked-source set and never push the aggregate to 2 on their own. Operators reading "exit 2" may still assume "nothing checked" when actually one source drifted + report is written. **Mitigation:** CLI stdout status line enumerates each source's status explicitly ("type-ownership: drift (3), helper-registry: parse-error, topology: clean, naming: skipped"). Reviewers read stdout + JSON, not just exit code.
- **P5-4-R5: `--strict-check-canon` combines with `--canon-draft`.** Both flags in one invocation emit independent manifest blocks; exit code is max of both. **Mitigation:** documented in §4.4 flag matrix + tested.
- **P5-4-R6: Canon §3 outliers absence is ambiguous.** A canon with no §3 could mean "no outliers ever" OR "renderer omitted because zero outliers". Parser treats absence as "zero outliers"; fresh side computes actual outliers. If both sides read empty, drift count is 0 (clean). If canon absent + fresh has outliers → all become `new-outlier-introduced`. **Mitigation:** intentional — matches P3-4 renderer's "§3 omitted when zero" convention. Integration test pins this as expected behavior.
