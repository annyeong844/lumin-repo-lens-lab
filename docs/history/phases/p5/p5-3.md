### 4.5 Category dispatch (differ algorithm)

Topology-drift has three sub-diffs that run independently and merge output:

**Sub-diff 1 ‚Äî Submodules:**
- Set of canon submodule-paths vs set of fresh submodule-paths.
- Fresh-only ‚Üí `submodule-added` (identity = submodule path).
- Canon-only ‚Üí `submodule-removed`.
- Both sides:
  - **`canon.sccMember` is sourced from ¬ß1 `SCC` column only.** Per canon-drift.md v1.1 ¬ß5.c ¬ß1, the ¬ß1 column is authoritative and ¬ß3 cycle membership MUST agree with it. Loader emits `canon-parse-error` when ¬ß1 and ¬ß3 disagree (PF-8) ‚Äî engine never sees the disagreeing state. This removes the reviewer-flagged ambiguity where `cyclic-submodule` label alone could have over-claimed membership.
  - If `canon.sccMember !== fresh.sccMember` ‚Üí `scc-status-changed` (identity = submodule path). `fresh.sccMember` comes from `collectTopologyStructure`'s `submodulesByPath[path].sccMember` (derived from `topology.json.sccs[]`).
  - If `canon.sccMember === fresh.sccMember` but `canon.label !== fresh.label` ‚Üí **no record in v1.** Degree-only label drift (e.g. `shared-submodule` ‚Üî `leaf-submodule` because in/out counts changed) is not a v1 category. Acceptable future canonical extension; v1 stays silent to avoid overclaim.
- `scc-status-changed` precedence rule (PF-8) therefore collapses to: emit IFF the SCC boolean differs. No label-only path.

**Sub-diff 2 ‚Äî Oversize files:**
- Set of canon oversize-file-paths vs set of fresh oversize-file-paths.
- Fresh-only (file is newly oversize) ‚Üí `oversize-changed` (identity = ownerFile path, carries `canon.label = null` / `fresh.label ‚àà {oversize, extreme-oversize}`).
- Canon-only (file dropped below threshold) ‚Üí `oversize-changed` (with `canon.label ‚àà {oversize, extreme-oversize}` / `fresh.label = null`).
- Both sides with differing label (tier change `oversize ‚Üî extreme-oversize`) ‚Üí `oversize-changed` (both labels populated).

**Sub-diff 3 ‚Äî Cross-edges (display-scope):**
- Set of canon ¬ß2 edges `{from, to}` vs set of fresh top-30 edges.
- Fresh-only ‚Üí `cross-edge-added` (identity = `<from> ‚Üí <to>` literal).
- Canon-only ‚Üí `cross-edge-removed`.
- Count delta within both sides ‚Üí NOT a category in v1 (no `cross-edge-count-delta` in canon-drift.md ¬ß3.1; ignored).

**Fresh top-30 derivation rule (¬ß4.5.4 ‚Äî Finding #4, 2026-04-22 pre-implementation):** the engine MUST construct the fresh top-30 snapshot by explicit sort, not by trusting producer-emitted ordering. Algorithm:

```
freshTop30 = (topology.crossSubmoduleTop ?? topology.crossSubmoduleEdges ?? [])
  .slice()                                        // copy before sort
  .sort((a, b) => (b.count - a.count)             // count desc
                 || (a.from < b.from ? -1 : a.from > b.from ? 1 : 0)
                 || (a.to < b.to ? -1 : a.to > b.to ? 1 : 0))
  .slice(0, 30);
```

Rationale: `topology.json` producers in the wild may emit `crossSubmoduleTop` already sorted, or emit `crossSubmoduleEdges` unsorted. A fixture or an external tool's output that is NOT count-desc-sorted would cause the engine's "top-30" to be the wrong top-30 ‚Äî silently mis-identifying which edges are in the display window. Explicit sort is cheap and removes the coupling. Stable tiebreak on `from` then `to` ensures identity assembly is deterministic. Step 0 test pin: engine receives an intentionally-shuffled `crossSubmoduleEdges` array and emits cross-edge records only for the actually-top-30-by-count set.

**Drift record assembly:** each sub-diff produces records via `makeDriftRecord({ kind: 'topology-drift', category, identity, canon, fresh })`. Family tag is auto-attached from `CATEGORY_TO_FAMILY` (already has all 6 entries).

## 8. Known risks

- **P5-3-R1: Cross-edge detection display-scope limitation (canonically documented in v1.1).** v1 compares canon ¬ß2 (top-30 display per canon-drift.md v1.1 ¬ß5.c ¬ß2) vs fresh top-30 (after explicit count-desc sort per ¬ß4.5.4). Edges that drift in or out of the top-30 window without being absolute add/remove are surfaced as cross-edge-added/removed records. **Mitigation:** (1) `Display scope: top-30` column on every cross-edge MD row + ¬ß4.4 narrative pointing at the canonical contract. (2) Diagnostic `kind: 'cross-edge-detection-is-display-scope'` optionally attached to every cross-edge drift record (advisory only; not a parse-error). (3) Future canonical v1.2+ revision could extend ¬ß5.c ¬ß2 to full-list and unlock absolute-scope detection.
- **P5-3-R-SORT: Fresh top-30 ordering cannot be trusted from producer.** `topology.json.crossSubmoduleTop` may be sorted, or may not be; `topology.json.crossSubmoduleEdges` is arbitrary-order. The engine MUST sort count-desc with `from`/`to` ASCII tiebreak before slicing (¬ß4.5.4). **Mitigation:** Step 0 test pin with a shuffled 31-edge input validates the sort is load-bearing; relying on producer ordering is explicitly forbidden by spec.
- **P5-3-R2: `topology.json` staleness under rapid refactors.** Unlike symbols.json (regenerated per `build-symbol-graph.mjs` run), topology.json comes from `measure-topology.mjs` which is a SEPARATE producer. Operator may forget to re-run topology producer after a structural refactor. **Mitigation:** same stale-freshness warning pattern as P5-1 ‚Äî compare `topology.json` mtime against source file mtimes; stderr warning when stale. Ship in v1.
- **P5-3-R3: `triage.json` mode detection edge cases.** When `triage.json` is absent, `collectTopologyStructure` runs single-package mode. On a real monorepo this under-classifies workspace boundaries but does NOT affect drift categories (workspace table is informational in v1). **Mitigation:** document in ¬ß5.c ¬ß5 notes; P5-4 or Phase 6+ may escalate.
- **P5-3-R4: Multi-section parser fragility.** ¬ß3 cycle listing has free-form subheadings (`### Cycle N (size M) ‚Äî forbidden-cycle ‚ùå`) that are more structural than tabular. A future P3-3 renderer change could break the parser silently. **Mitigation:** round-trip test in Step 0 uses `renderCyclesSection` output fixtures (reuse P3-3's real renderer for the fresh-side input; hand-write the canon-side input to match exact format); any renderer shift breaks the round-trip immediately.
- **P5-3-R5: SCC identity granularity.** A 4-member cycle produces 4 `scc-status-changed` records. On repos with many cycles, record count can balloon. **Mitigation:** accept for v1 ‚Äî per-member granularity matches canon-drift.md ¬ß4 identity contract. Aggregation at cycle level is a future category.
- **P5-3-R6: `oversize-changed` when both labels are null.** The "both sides populated and tier differs" sub-case is straightforward; the one-sided cases (fresh becomes oversize, or canon-oversize disappears from fresh) need explicit null handling in the record's `canon.label` or `fresh.label`. **Mitigation:** renderer inserts `‚Äî` for null label; integration test pins this presentation.
