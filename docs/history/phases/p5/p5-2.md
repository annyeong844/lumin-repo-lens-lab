### 4.4 No owner-changed-style upgrade

Unlike P5-1's type-drift (which upgrades 1:1 same-name-different-file into `owner-changed`), P5-2 does **NOT** upgrade same-name-different-file helper pairs. Reason: canon-drift.md §3.1 does not include a `helper-owner-changed` category. Behavior:

- Same `exportedName` in canon + fresh, different `ownerFile` → emits one `helper-removed` (canon side) + one `helper-added` (fresh side). Neither record carries a `confidence: 'low'` flag by default — they are crisp removals and additions from the identity-diff perspective.
- Operator reviewer interprets manually. If the pattern recurs often, reviewer can request canonical extension (canon-drift.md v1.1 adds `helper-owner-changed`); P5-2 v2 would then add the upgrade.

### 4.5 Label-change category dispatch

When `canon.label !== fresh.label` at the same identity, the differ must pick exactly one of three label-change-family categories. Dispatch order (first match wins):

1. **`contamination-changed`** (family: `content-shifted`) — **evidence-gated PER-IDENTITY.** Fires IFF both of the following hold:
   - `symbols.helperOwnersByIdentity[identity] != null` for the SPECIFIC drift identity (not a repo-level flag — a single unrelated entry in `helperOwnersByIdentity` does NOT authorize `contamination-changed` for a different identity), AND
   - EITHER `canon.label` OR `fresh.label` ∈ `{severely-any-contaminated-helper, ANY_COLLISION_HELPER}`.

   When the label transition involves a contamination label but the drift identity lacks a `helperOwnersByIdentity` entry, the engine MUST NOT emit `contamination-changed`. The transition falls through to rule 3 (`label-changed`). When NO identity in the run has enrichment, a run-level advisory diagnostic `helper-contamination-enrichment-unavailable` is also attached at engine entry (§4.6). This preserves Iron Law: a non-contamination `fresh.label` without per-identity capability evidence means "the collector didn't observe contamination facts for THIS helper", NOT "contamination is gone".
2. **`fan-in-tier-changed`** (family: `label-changed`) — if BOTH `canon.label` AND `fresh.label` are in `{zero-internal-fan-in-helper, shared-helper, central-helper}`. These three labels correspond directly to `classifyHelperIdentity`'s fan-in tier thresholds (0 / 1-2 / ≥3). NOT evidence-gated by `helperOwnersByIdentity` — fan-in is always grounded from the AST resolver pass (§4.5 fan-in groundedness note below).
3. **`label-changed`** (family: `label-changed`) — fallback. Covers: transitions involving `HELPER_DUPLICATE_STRONG` / `HELPER_DUPLICATE_REVIEW` / `HELPER_LOCAL_COMMON` / `low-signal-helper-name`; contamination-label transitions downgraded from rule 1 when the drift identity lacks a `helperOwnersByIdentity` entry; any other pair not caught by rules 1 or 2.

Static enumeration to pin in the engine (and mirror in a unit test):

```js
// _lib/check-canon-helpers.mjs local constants
const CONTAMINATION_HELPER_LABELS = new Set([
  'severely-any-contaminated-helper',
  'ANY_COLLISION_HELPER',
]);
const FAN_IN_TIER_HELPER_LABELS = new Set([
  'zero-internal-fan-in-helper',
  'shared-helper',
  'central-helper',
]);
```

**Evidence-gated dispatch** (reviewer Finding #3, 2026-04-22 — pre-implementation; refined to per-identity by reviewer Finding #1, 2026-04-22 — post-landing): contamination-changed requires PER-IDENTITY capability evidence on the fresh side. Without a `symbols.helperOwnersByIdentity[identity]` entry for THIS specific drift identity, the engine has no grounded way to assert "fresh is clean" for THIS helper — a `fresh.label === 'central-helper'` reading without per-identity enrichment means "fresh collector did not observe contamination facts for THIS helper", NOT "contamination is cleaned up". Claiming the latter is an evidence overreach (Iron Law).

Dispatch revision (strict order):

```
IF canon.label !== fresh.label:
  IF canon.label OR fresh.label ∈ CONTAMINATION_HELPER_LABELS:
    IF symbols.helperOwnersByIdentity[identity] != null:
      → `contamination-changed`
    ELSE:
      → `label-changed`  (per-identity evidence-gated downgrade)
      + run-level advisory diagnostic `helper-contamination-enrichment-unavailable`
        attached at engine entry (§4.6) when NO identity has enrichment
  ELSE IF canon.label AND fresh.label ∈ FAN_IN_TIER_HELPER_LABELS:
    → `fan-in-tier-changed`
  ELSE:
    → `label-changed`
```

Consequence: with today's producer (`helperContamination === 'unavailable'`), `contamination-changed` NEVER fires. It becomes reachable when a future producer revision emits `symbols.helperOwnersByIdentity`. This is the honest state; fixture `canon-drift-helpers-contamination-changed` must ship with `symbols.helperOwnersByIdentity` populated (§5.6) or the dispatch will downgrade to `label-changed`.

### 4.6 Collector diagnostic promotion policy (reviewer Finding #1)

`collectHelperIdentities` pushes entries into its `diagnostics` array during the fresh pass. P5-2's engine MUST inspect these and promote per-kind:

- **`kind: 'parse-error'`** (e.g. `reason: 'extractor-threw'` — oxc-parser failed on a file) → **source-level parse-error**. The engine returns `status: 'parse-error'`, empty drifts, `reportMarkdown: null`, and the diagnostic is surfaced in `perSource["helper-registry"].diagnostics`. Exit code 2. JSON write-anyway per P5-1 policy. MD skipped.
  - Rationale: partial fresh set risks false `helper-removed` records (a file whose AST failed to parse contributes zero exports, making its helpers appear "removed" from canon). "Clean" or a spurious drift report on partial evidence is the worst failure mode, parallel to canon-drift.md §5.e "lenient partial parse is explicitly forbidden".
  - Test pin: engine test where `extractFn` throws for one file among many → `status === 'parse-error'`, no helper-removed records for that file, diagnostic carries the original reason + file path.
- **`kind: 'call-graph-cross-check'`** → **advisory only.** Passes through to `perSource.diagnostics` but does NOT promote status. Engine continues with the current fresh set.
- **`kind: 'helper-contamination-enrichment-unavailable'`** (synthetic; set by the engine at RUN LEVEL when NO identity has `symbols.helperOwnersByIdentity` entries — i.e. `meta.helperContamination !== 'available'`) → **advisory.** Does NOT promote status. Per-identity gating in §4.5 is the authoritative dispatch rule; this diagnostic is a run-wide hint that the producer has emitted zero enrichment.
- **Any other `kind`** → advisory by default; non-`parse-error` diagnostics never promote status. Engine changelog note: if a new diagnostic kind is introduced by `collectHelperIdentities` in the future, reviewer must classify it (fatal vs advisory) in a single-commit edit alongside the collector change.

**Fan-in groundedness note** (reviewer P1-1, 2026-04-22): `fan-in-tier-changed` is safe even when `callGraph === null`. `collectHelperIdentities` computes fan-in directly from the AST resolver pass (`_lib/canon-draft-helpers.mjs:197` — `def.fanIn = distinctConsumerFiles.get(identity)?.size ?? 0`). `call-graph.json` is cross-check only; its absence/corruption produces diagnostics but never changes `def.fanIn`. Therefore fan-in-tier dispatch is grounded under both `callGraph = null` and `callGraph = <stale>`.

### 4.7 Optional-artifact corrupt semantics (reviewer Finding #3)

Both `symbols.json` and `call-graph.json` are OPTIONAL for helper-registry dispatch. CLI loads them via `loadIfExists` (non-strict) — same as `generate-canon-draft.mjs --source helper-registry`. Behavior:

- **symbols.json absent** → `symbols=null`, enrichment unavailable, advisory diagnostic emitted. Run continues.
- **symbols.json corrupt JSON** → `loadIfExists` returns null + logs parse failure to stderr. Treated identically to absent (enrichment unavailable, advisory diagnostic). Run continues, exit normally based on drift/clean status.
- **call-graph.json absent** → `callGraph=null`, cross-check skipped. stderr warning. Run continues.
- **call-graph.json corrupt JSON** → `loadIfExists` returns null + logs parse failure to stderr. Treated identically to absent. Run continues.
- **No `--strict-freshness` hard-fail in v1.** A future `--strict-enrichment` could escalate symbols/call-graph corruption to exit 2; not in scope for P5-2.

Rationale: helpers' primary signal is the fresh AST pass. enrichment / cross-check are advisory; corruption in those artifacts must not abort the run. This is a different policy from P5-1's type-ownership path (where symbols.json IS the primary signal and corruption = exit 2).

## 8. Known risks

- **P5-2-R1: `helper-owner-changed` absence may surface as noise.** Helpers moving files (refactors) emit both `helper-removed` AND `helper-added`. For a repo with many such moves, the noise-to-signal ratio dips. **Mitigation:** v1 accepts the noise; canon-drift.md §3.3 "additive extension policy" allows a v1.1 canonical edit adding `helper-owner-changed`. If reviewer observes this risk materializing, escalate to canonical revision.
- **P5-2-R2: Fresh-AST cost.** `collectHelperIdentities` walks every file returned by `collectFiles`, extracts via oxc-parser, resolves specifiers. On a 1000-file repo this is ~5-15s (same budget as session.md v2 §7 P5-1-R4 noted). **Mitigation:** acceptable for dev/CI flow. Incremental cache (parallel to `_lib/incremental.mjs`) is v2+.
- **P5-2-R3: call-graph.json staleness propagation.** `collectHelperIdentities` uses call-graph for cross-check; if stale, diagnostics may under-report. P3-2's renderer emits a `> ⚠ Source call-graph.json is <N> hours stale` header warning. **Mitigation:** P5-2 inherits the same staleness detection by passing `nowMs` to the collector; stderr warning when applicable. Does not change the drift result, just the diagnostic footnote.
- **P5-2-R4: Contamination detection requires fresh-side capability.** `collectHelperIdentities` reads `symbols.helperOwnersByIdentity` to annotate fresh-side contamination; `build-symbol-graph.mjs` does NOT emit this field in v1 (declares `meta.supports.anyContamination: false`). Ordinary runs therefore have `meta.helperContamination === 'unavailable'`. Per the **evidence-gated dispatch** in §4.5:
  - Transitions involving contamination labels under `'unavailable'` downgrade to `label-changed`, NOT `contamination-changed`.
  - `contamination-changed` is NOT emitted.
  - The advisory diagnostic `kind: 'helper-contamination-enrichment-unavailable'` is surfaced in `perSource["helper-registry"].diagnostics`.
  - This is honest — a `fresh.label` of `central-helper` under unavailable capability means "the collector did not observe contamination facts", not "contamination is cleaned up". Claiming the latter would violate the Iron Law.

  `contamination-changed` is emitted ONLY when fresh-side helper-contamination capability is available (fixture or future producer supplies `symbols.helperOwnersByIdentity`). **Mitigations already in place:** (1) advisory diagnostic on every unavailable run (§4.6). (2) Fixture `canon-drift-helpers-contamination-changed` ships `symbols.helperOwnersByIdentity` to activate the available branch (§5.6). (3) Full ordinary-run detection lands when a future producer revision emits `helperOwnersByIdentity` — tracked as an out-of-P5 prerequisite.
- **P5-2-R5: Label-dispatch rule ambiguity.** If a future §10.3 label is added that straddles the "contamination" and "fan-in tier" semantic axes, the dispatch order may need revision. **Mitigation:** §4.5's first-match-wins order is documented + test-pinned; any new label requires a canon-drift-md revision and a P5-2 engine update in lockstep.
- **P5-2-R6: Row-count growth.** Adding fan-in to multiple MD columns plus signal evidence increases report width. **Mitigation:** accept — fan-in is load-bearing for helper-drift analysis; truncating would lose evidence.
