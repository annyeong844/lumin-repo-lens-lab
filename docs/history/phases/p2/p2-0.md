## 4. Output shapes

### 4.0 escapeKind de-duplication precedence

Several escapeKinds share the same underlying `TSAnyKeyword` AST node. Without a precedence rule, one occurrence would emit multiple facts and inflate the delta. Authoritative precedence (specific beats generic):

| Winner | Over | Example |
|---|---|---|
| `rest-any-args` | `explicit-any` | `(...args: any[]) => {}` — ONE fact with kind `rest-any-args`, NOT also `explicit-any` |
| `index-sig-any` | `explicit-any` | `{ [k: string]: any }` — ONE fact with `index-sig-any` |
| `generic-default-any` | `explicit-any` | `type Box<T = any>` — ONE fact with `generic-default-any` |
| `as-unknown-as-T` | `as-any` | `x as unknown as Foo` — ONE fact with `as-unknown-as-T` (NOT also `as-any`) |
| `angle-any` | `explicit-any` | `<any>expr` (legacy cast) — ONE fact with `angle-any` |

The extractor picks the MOST SPECIFIC kind for each occurrence. `test-extract-ts-escapes.mjs` pins each precedence case with a "emits exactly one fact" assertion — two facts from the same occurrence is a defect.

### 4.1 `type-escape` fact (post-amendment)

Per `canonical/fact-model.md §3.9` after Step 0:

```json
{
  "file": "src/api/client.ts",
  "line": 42,
  "escapeKind": "as-any",
  "codeShape": "response as any",
  "normalizedCodeShape": "response as any",
  "insideExportedIdentity": "src/api/client.ts::fetchUser",
  "occurrenceKey": "sha256:<64-hex>"
}
```

Key rules:

- `line` — 1-based. Display / citation only; NEVER used for matching.
- `codeShape` — original source slice (preserved verbatim including whitespace). For comment-style escapes (`ts-ignore` / `ts-expect-error` / `no-explicit-any-disable`), carries the FULL comment text (e.g. `"@ts-expect-error upstream type bug"`).
- `normalizedCodeShape` — `codeShape` with inner whitespace runs collapsed to single spaces AND trailing `;` dropped. Purely for matching; NOT for display. **Implementation constraint:** do NOT normalize by regex over the raw source. Normalize token-aware or AST-slice-aware so whitespace INSIDE string / template literals is preserved verbatim. `foo as "a   b" as any` normalizes to `foo as "a   b" as any` (the `a   b` interior stays intact), NOT `foo as "a b" as any`. A pinning test covers the string-literal case.
- `insideExportedIdentity` — `"ownerFile::exportedName"` when the occurrence is lexically inside an exported function / class / const; `null` when top-level or inside a non-exported scope. **Export-form resolution** (pinned in tests):

  | Source form | `insideExportedIdentity` value |
  |---|---|
  | `export function fetchUser() { ... }` | `<file>::fetchUser` |
  | `export class UserModel { ... }` | `<file>::UserModel` |
  | `export const fetchUser = () => { ... }` | `<file>::fetchUser` (const bound to an arrow / function expression / class expression) |
  | `function fetchUser() {} ... export { fetchUser }` | `<file>::fetchUser` (producer walks export specifiers back to the local declaration) |
  | `function foo() {} ... export { foo as bar }` | `<file>::bar` — use the EXPORTED name, not the local one (matches identity-and-alias.md §2 `ownerFile::exportedName` rule) |
  | `export default function fetchUser() {}` | `<file>::default` — named but default-exported |
  | `export default () => { ... }` | `<file>::default` — anonymous |
  | `export default class {}` | `<file>::default` |
  | local helper (`function unused() {}`, not exported) | `null` |
  | top-level escape outside any function/class body | `null` |
  | nested non-exported function inside an exported one | innermost-enclosing EXPORTED identity wins (the exported parent) |

  The last rule (`nested non-exported inside exported parent`) keeps the matching anchor on the exported surface — where Claude's planned-escape `locationHint` would point. `_lib/extract-ts-escapes.mjs` walks outward from the occurrence's AST path and takes the first exported binding it finds; if none, `null`.
- `occurrenceKey` — `sha256(file + '|' + escapeKind + '|' + normalizedCodeShape + '|' + (insideExportedIdentity ?? '<top-level>'))`. Stable across formatter runs that shift lines but leave code shape intact.

### 4.2 `any-inventory.json` artifact shape

```json
{
  "meta": {
    "tool": "any-inventory.mjs",
    "generated": "2026-04-20T12:30:00Z",
    "complete": true,
    "scope": "TS/JS production files",
    "includeTests": false,
    "exclude": ["node_modules", "dist"],
    "fileCount": 123,
    "supports": {
      "typeEscapes": true,
      "escapeKinds": [
        "explicit-any", "as-any", "angle-any", "as-unknown-as-T",
        "rest-any-args", "index-sig-any", "generic-default-any",
        "ts-ignore", "ts-expect-error", "no-explicit-any-disable"
      ]
    }
  },
  "typeEscapes": [ /* one entry per occurrence, shape per §4.1 */ ]
}
```

`meta.supports.escapeKinds` MUST equal the canonical §3.9 set exactly. Order-sensitive — bootstrap drift test parses canonical markdown and asserts array equality (same elements, same order).

**`meta.complete` rule:**

```
meta.complete = (meta.filesWithParseErrors.length === 0)
```

`complete: true` is the producer's promise that EVERY file in the scanned range parsed successfully AND every occurrence within them is listed. A single parse-error flips `complete` to `false` — the inventory is NOT "complete" if it silently skipped files. P2-1 sees `complete: false` and degrades the per-occurrence delta to `[확인 불가, reason: any-inventory incomplete due to parse errors]` for the files whose shape matches the parse-error set, preserving confidence for files outside that set.

`meta.filesWithParseErrors` shape (populated when any file failed to parse):

```json
[
  { "file": "src/bad.ts", "message": "Unexpected token", "line": 12 }
]
```

File-relative paths, concise parser error message (capped at ~200 chars), line when available.

## 8. Known risks

- **`_lib/extract-ts-escapes.mjs` completeness.** Missing a single escapeKind means silent-new misses real escapes. Mitigation: the 15-case unit test covers all 10 kinds + edge cases; the drift test pins `escapeKinds[]` to canonical; integration test in P2-2 catches end-to-end.
- **Normalization ambiguity.** `normalizedCodeShape` collapsing "inner whitespace runs" — what about string literals inside the code slice (e.g. `foo as "a   b" as any`)? The spec is: collapse ONLY whitespace outside string literals. If the implementation shortcut collapses whitespace inside strings, shared-codeShape matching becomes lossy. Pinning test includes a string-literal case.
- **`occurrenceKey` collision probability.** sha256 over structural attributes → collision is negligible for realistic repos. If two legitimately-different occurrences end up with the same key (same file, escapeKind, normalizedCodeShape, insideExportedIdentity), they're semantically indistinguishable — the matching layer treating them as one is acceptable.
- **Hook ordering in `pre-write.mjs`.** Inserting the inventory run BEFORE lookups/render could mask lookup errors; AFTER lookups/render could make the advisory artifact reference a missing file if hook fails mid-way. Spec pins: hook runs AFTER lookups are populated BUT BEFORE `writeAdvisory` — so the advisory object (in-memory) is stamped with a valid `preWrite.anyInventoryPath` only if the inventory was actually written successfully. On hook failure, `anyInventoryPath` stays absent and `failures[]` records the error.
- **Canonical amendment coupling.** Step 0 edits a canonical file. Per P2 §6, canonical edits need explicit motivation. P2 §4.1 + this file's §5.1 document the motivation (`occurrenceKey` / `normalizedCodeShape` are load-bearing for the delta engine's stability). Reviewer approves the amendment as part of P2 phase approval — already granted.

## 10. Handoff to P2-1

P2-1 consumes:

- `any-inventory.json` produced by `any-inventory.mjs` as both `beforeInventory` and `afterInventory`.
- `pre-write-advisory.<invocationId>.json.preWrite.anyInventoryPath` → points at the `before` snapshot for a given invocation.

- `meta.supports.escapeKinds[]` order is the order consumers iterate; P2-1 render must not reorder.
