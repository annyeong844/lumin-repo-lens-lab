## 4. Output shapes

### 4.1 `computeDelta` signature + result

```ts
computeDelta({
  preWriteAdvisory,      // object: P1 advisory artifact (JSON)
  beforeInventory,       // object | null: any-inventory.json at pre-write time
  afterInventory,        // object | null: any-inventory.json at post-write time
  deltaInvocationId,     // string: CALLER-INJECTED. Generated outside computeDelta so the
                         //         function stays pure (same inputs → byte-identical output).
                         //         post-write.mjs calls generateDeltaInvocationId() and passes it in.
}) → DeltaResult

type DeltaResult = {
  preWriteInvocationId: string,         // copied from preWriteAdvisory.invocationId — pairs the delta with the advisory
  deltaInvocationId: string,            // passed through from input — pairs the delta with its specific file
  intentHash: string,                   // copied from preWriteAdvisory.intentHash
  baseline: {
    status: 'available' | 'missing',
    source: string | null,              // preWriteAdvisory.preWrite.anyInventoryPath when available
    reason?: string,                    // populated when status === 'missing'
  },
  capabilityParity: {
    status: 'ok' | 'mismatch' | 'unchecked' | 'missing',
    //   ok         — afterInventory usable; capability claim matches canonical
    //   mismatch   — afterInventory present but meta.supports.typeEscapes !== true
    //                OR meta.supports.escapeKinds drifts from canonical §3.9
    //   unchecked  — reserved for future capability sources; P2-1 v1 never emits this
    //   missing    — afterInventory absent entirely (e.g. --no-fresh-audit + no prior file)
    mismatchDetail?: string,
  },
  scanRangeParity: {
    status: 'ok' | 'mismatch' | 'baseline-missing',
    //   mismatch invalidates ONLY the before/after comparison.
    //   Planned matching (advisory.intent × afterInventory) stays valid.
    mismatchDetail?: string,
  },
  inventoryCompleteness: {
    //   P2-0 §4.2 contract: meta.complete === false means some files had parse errors
    //   and their occurrences are NOT in typeEscapes[]. P2-1 records this so:
    //     (a) downstream consumers see the blind spot in JSON artifact, and
    //     (b) renderer knows not to emit a "clean" summary when afterComplete is false.
    afterComplete: boolean,             // = afterInventory?.meta?.complete === true
    beforeComplete: boolean | null,     // null when beforeInventory missing/unusable; else meta.complete
    filesWithParseErrors: Array<{
      side: 'before' | 'after',
      file: string,
      message: string,
      line: number,
    }>,
  },
  entries: Array<DeltaEntry>,           // one per matched occurrence + each unmatched silent-new/observed-unbaselined
  summary: {
    planned: number,
    plannedNotObserved: number,
    silentNew: number,
    preExisting: number,
    removed: number,
    observedUnbaselined: number,
  },
  failures: Array<{kind: string, reason: string}>,  // non-fatal issues: malformed inputs, after-inventory-missing, etc.
}

type DeltaEntry = {
  label: 'planned' | 'planned-not-observed' | 'silent-new' |
         'pre-existing' | 'removed' | 'observed-unbaselined',
  escapeKind: string,                    // one of canonical §3.9
  file: string | null,                   // null only for planned-not-observed when locationHint was "unknown"
  line: number | null,
  codeShape: string | null,              // null for planned-not-observed (nothing observed)
  normalizedCodeShape: string | null,
  insideExportedIdentity: string | null,
  occurrenceKey: string | null,          // null for planned-not-observed
  plannedEntry: {                        // present on 'planned' and 'planned-not-observed'
    escapeKind: string,
    locationHint: string,
    codeShape?: string,
    reason: string,
    alternativeConsidered?: string,
  } | null,
  diagnostics: string[],                 // e.g. ['ambiguous-planned-match']; empty array by default
}
```

**Purity contract (reviewer P0-1 fix).** `computeDelta` MUST be deterministic:
- Same `(preWriteAdvisory, beforeInventory, afterInventory, deltaInvocationId)` → byte-identical `DeltaResult`.
- No `new Date()`, `Date.now()`, `Math.random()`, `crypto.randomBytes()`, `process.*`, or filesystem calls inside `computeDelta`.
- `deltaInvocationId` is the only "freshly-generated" value; the caller (`post-write.mjs` or test) owns its generation.
- Pinning test: call `computeDelta` twice with the same inputs (including deltaInvocationId) and assert `JSON.stringify(resultA) === JSON.stringify(resultB)`.

**Note on `diagnostics[]`.** `ambiguous-planned-match` can appear on `silent-new`, `pre-existing`, OR `observed-unbaselined` entries — whichever label baseline comparison assigns. It is NOT an independent status that forces `silent-new`. See §4.2.

### 4.2 Planned ↔ observed matching (authoritative)

Implemented as a pure function inside `_lib/post-write-delta.mjs`. Runs BEFORE baseline comparison so planned-status wins; baseline comparison then labels the remaining observed occurrences.

`beforeByKey` is built ONCE at the top of classification (before §4.2) and shared with §4.3 — two Map builds waste work.

5-step rule:

```
// Shared state (built once per computeDelta call).
beforeByKey = baseline.status === 'available'
  ? new Map(beforeInventory.typeEscapes.map(e => [e.occurrenceKey, e]))
  : new Map();

for each plannedEscape in preWriteAdvisory.intent.plannedTypeEscapes:
  candidates = afterInventory.typeEscapes where escapeKind === plannedEscape.escapeKind
  candidates = candidates filter location match:
    - observed.insideExportedIdentity === planned.locationHint, OR
    - observed.file === planned.locationHint, OR
    - planned.locationHint.endsWith('/') && observed.file.startsWith(planned.locationHint), OR
    - planned.locationHint === "unknown"  (wildcard for unspecified location)
  if candidates.length === 0:
    → emit 'planned-not-observed' entry with plannedEntry but null occurrence fields
    continue

  // (P1-1 reviewer fix) Absent-from-before preference.
  // When baseline is trustworthy (available AND scanRangeParity ok), prefer
  // candidates that are not already in beforeInventory — planned escapes are
  // about to be added, so "new" candidates are the natural match.
  if baseline.status === 'available' AND scanRangeParity.status === 'ok':
    newCandidates = candidates.filter(c => !beforeByKey.has(c.occurrenceKey))
    if newCandidates.length >= 1:
      candidates = newCandidates
    // else: every candidate is pre-existing; fall through — planned matches the
    //       first deterministic pre-existing candidate (the planned declaration
    //       happened to describe code that was already there; honest pick).

  // Code-shape tiebreak.
  if planned.codeShape is set and candidates.length >= 2:
    exactMatches = candidates where normalizedCodeShape === normalizeCodeShape(planned.codeShape)
    //                                ^^^^^^^^^^^^^^^^ imported from _lib/extract-ts-escapes.mjs
    if exactMatches.length >= 1:
      candidates = exactMatches

  // One-to-one.
  choice = candidates sorted by (file, line, occurrenceKey) [0]
  mark choice as matched-to-this-planned
  → emit 'planned' entry with plannedEntry + choice's fields

  // Ambiguity — REMAINDER IS NOT HARD-LABELED 'silent-new'.
  if candidates.length >= 2 after tiebreak:
    for each remaining candidate beyond choice:
      attach diagnostic 'ambiguous-planned-match' to it (carry-over)
      // leave it UNMARKED so baseline comparison determines its honest label.
      // baseline-present: remainder → pre-existing (if key in before) OR silent-new
      // baseline-missing: remainder → observed-unbaselined
      // In all three cases the 'ambiguous-planned-match' diagnostic travels with it.
```

After matching, every `afterInventory.typeEscape` is either:
- marked `planned` (matched one-to-one to a plannedEscape), OR
- unmarked (possibly carrying `ambiguous-planned-match` diagnostic) → proceeds to baseline comparison.

**Why absent-from-before preference.** With `locationHint: "unknown"` (wildcard) and baseline available, if multiple candidates share an escapeKind, the planned escape is almost certainly the ones that just came into existence. Preferring absent-from-before prevents the subtle bug where a planned declaration gets pointed at a pre-existing occurrence while the actually-new one falls through to silent-new. Fallback behavior when ALL candidates are pre-existing: honest pick of the first — a planned declaration describing existing code is unusual but legal, and the render shows "planned and observed" correctly.

**Why remainder passes through, not hard silent-new.** Ambiguity means "we couldn't prove which one was planned"; it does NOT mean "this is new." If the remainder existed in the before-snapshot, it is pre-existing — the ambiguity is orthogonal to whether it is new. Hard-labeling the remainder `silent-new` would produce false Stage 3 acknowledgement requirements, violating the "no false silent-new claim" discipline.

### 4.3 Baseline comparison (for unmatched occurrences)

```
if baseline.status === 'missing':
  for each unmarked after in afterInventory:
    carryDiagnostics = after.diagnostics ?? []
    → 'observed-unbaselined' with diagnostics: carryDiagnostics
  // 'removed' NOT emitted (cannot compute without before)
else:
  afterByKey = new Map(afterInventory.typeEscapes.map(e => [e.occurrenceKey, e]))
  // beforeByKey already built at the top of classification (§4.2 shared state).

  for each unmarked after in afterInventory:
    carryDiagnostics = after.diagnostics ?? []
    if beforeByKey.has(after.occurrenceKey):
      → 'pre-existing' with diagnostics: carryDiagnostics
    else:
      → 'silent-new' with diagnostics: carryDiagnostics

  for each before in beforeInventory where NOT afterByKey.has(before.occurrenceKey):
    → 'removed'
```

`requiredAcknowledgements(delta)` filters `delta.entries.filter(e => e.label === 'silent-new')` — NEVER any other label, EVEN when the entry carries `ambiguous-planned-match`. Pinned by test.

### 4.4 Capability parity + scan-range parity

Both checks run BEFORE classification so the delta result tells the truth about confidence. Neither check mutates any input; they change which labels the classification loop is allowed to emit.

**Capability source.** The authoritative capability signal is **the inventory artifact itself**, specifically `meta.supports.typeEscapes`, `meta.supports.escapeKinds[]`, and `meta.complete`. `preWriteAdvisory.capabilities` is the P1-0 symbols.json projection (derived from `symbols.json.meta.supports`); P2-1 does NOT use it for type-escape capability decisions.

Helper:

```js
function inventoryUsable(inv) {
  if (!inv) return false;
  if (inv.meta?.supports?.typeEscapes !== true) return false;
  const kinds = inv.meta?.supports?.escapeKinds ?? [];
  if (kinds.length !== CANONICAL_ESCAPE_KINDS.length) return false;
  for (let i = 0; i < kinds.length; i++) {
    if (kinds[i] !== CANONICAL_ESCAPE_KINDS[i]) return false;
  }
  return true;
}
// `meta.complete === false` is tolerated at the capability level; it surfaces
// via inventoryCompleteness (§4.1) and shapes the renderer summary (§4.5).
// Classification still runs for the files that DID parse.
```

Rules:

```
// Capability parity.
if !afterInventory:
  capabilityParity = {status: 'missing',
                      mismatchDetail: 'afterInventory absent — see failures[]'}
  failures += {kind: 'after-inventory-missing', reason: ...}
  → entries: []
  → requiredAcknowledgements returns []

else if !inventoryUsable(afterInventory):
  capabilityParity = {status: 'mismatch',
                      mismatchDetail: '<which specific check failed>'}
  failures += {kind: 'after-inventory-unusable', reason: ...}
  → entries: []
  → requiredAcknowledgements returns []

else if beforeInventory and !inventoryUsable(beforeInventory):
  capabilityParity = {status: 'ok'}
  baseline = {status: 'missing', source: advisory.preWrite?.anyInventoryPath ?? null,
              reason: 'beforeInventory present but unusable (meta.supports.typeEscapes !== true or escapeKinds drift)'}
  → unmarked observed entries become observed-unbaselined
  → requiredAcknowledgements returns []

else:
  capabilityParity = {status: 'ok'}

// Scan-range parity — only meaningful when baseline is usable.
if baseline.status === 'available':
  const excludeBefore = JSON.stringify([...(beforeInventory.meta.exclude ?? [])].sort());
  const excludeAfter  = JSON.stringify([...(afterInventory.meta.exclude  ?? [])].sort());
  //                                   ^^^ non-mutating spread-then-sort; pure computeDelta
  if (before.meta.scope !== after.meta.scope ||
      before.meta.includeTests !== after.meta.includeTests ||
      excludeBefore !== excludeAfter):
    scanRangeParity = {status: 'mismatch',
                       mismatchDetail: '<which field differs, with before vs after values>'}
    // IMPORTANT — mismatch invalidates the before/after comparison only.
    //   Planned matching STAYS intact.
    //   Absent-from-before preference in §4.2 is SKIPPED under mismatch
    //     (since "absent from before" is no longer a trustworthy signal).
    //   Unmarked observed entries degrade to 'observed-unbaselined' honestly.
    //   'removed' is NOT computed.
    //   requiredAcknowledgements returns [] (silent-new requires trustworthy
    //     baseline comparison).
  else:
    scanRangeParity = {status: 'ok'}
else:
  scanRangeParity = {status: 'baseline-missing'}

// Inventory completeness — always populated, regardless of capability/scan outcomes.
inventoryCompleteness = {
  afterComplete: afterInventory?.meta?.complete === true,
  beforeComplete: baseline.status === 'available'
                    ? (beforeInventory.meta?.complete === true)
                    : null,
  filesWithParseErrors: [
    ...(afterInventory?.meta?.filesWithParseErrors ?? []).map(e => ({...e, side: 'after'})),
    ...(baseline.status === 'available'
         ? beforeInventory.meta?.filesWithParseErrors ?? []
         : []).map(e => ({...e, side: 'before'})),
  ],
};
```

**Pinning:** Tests assert behavior is identical whether `preWriteAdvisory.capabilities` is absent, `null`, `{}`, or `{typeEscapes: false}` — as long as `afterInventory` is usable, classification proceeds. The advisory's `capabilities` block is a symbols projection, not a type-escape one.

### 4.5 Markdown render format

Matches `canonical/any-contamination.md` §6 Stage 2 output exactly:

```
## post-write delta (canonical/any-contamination §6 Stage 2)

Any delta (silent-new counts):
- explicit any:               +N
- as any:                     +N
- as unknown as T:            +N
- angle-any:                  +N
- rest-any-args:              +N
- index-sig-any:              +N
- generic-default-any:        +N
- ts-ignore:                  +N
- ts-expect-error:            +N
- no-explicit-any-disable:    +N

Planned and observed:
1. src/vendor/wrapper.ts:42  `response as unknown as ThirdPartyShape`
   insideExportedIdentity: src/vendor/wrapper.ts::adaptResponse
   planned? yes — upstream SDK lacks type exports

New escape sites (silent-new — REQUIRE acknowledgment):
1. src/api/client.ts:42  `response as any`
   insideExportedIdentity: src/api/client.ts::fetchUser
   planned? no — reason missing
   [grounded, any-inventory.json.typeEscapes[occurrenceKey=sha256:...] present in after, absent in before]

Observed without baseline:
1. src/legacy.ts:12  `payload: any`
   [확인 불가, reason: before-inventory missing; cannot determine new-vs-existing]

Planned but not observed:
1. planned `ts-expect-error` at `src/migration.ts::adapter` — not observed after write.

Removed:
1. src/old.ts:5  `legacy as any`  (present in before, absent in after)

Inventory completeness:
- after-inventory complete: {yes | no — N file(s) had parse errors}
- before-inventory complete: {yes | no — N file(s) had parse errors | n/a (baseline missing)}
- files with parse errors: {empty | list: side:file (line) — message}

Baseline status: {available | missing — with reason}
Capability parity: {ok | mismatch | missing — with detail}
Scan-range parity: {ok | mismatch | baseline-missing — with detail}

<summary line — see rules below>
```

**"Any delta" block semantics.** `+N` counts `silent-new` entries only — the acknowledgment-required portion. Other labels get their own sections and are NOT double-counted here:

| Label | Where rendered | Counted in "Any delta"? |
|---|---|---|
| `silent-new` | "New escape sites" section AND "Any delta" | Yes |
| `planned` | "Planned and observed" | No |
| `planned-not-observed` | "Planned but not observed" | No |
| `pre-existing` | not rendered (noise baseline; acknowledgment not required) | No |
| `observed-unbaselined` | "Observed without baseline" | No |
| `removed` | "Removed" | No |

Sections are omitted when empty (except "Any delta" which always renders all 10 kinds, and the Inventory / Baseline / Capability / Scan-range status lines which always render).

**Summary-line rules (reviewer P0-2 + P1-2 fix).** The closing summary line is NOT "No silent new any in the scan range" in all zero-silentNew cases — that phrasing is reserved for the *fully-clean* state:

| Conditions (ALL must hold) | Summary line |
|---|---|
| `silentNew === 0` AND `capabilityParity.status === 'ok'` AND `scanRangeParity.status === 'ok'` AND `baseline.status === 'available'` AND `inventoryCompleteness.afterComplete === true` AND `inventoryCompleteness.beforeComplete === true` | `No silent new any in the scan range.` |
| `silentNew === 0` but any of the above conditions fails | `No silent-new acknowledgements required; delta confidence is limited by <reason-list>.` |
| `silentNew > 0` | `silent-new — REQUIRE acknowledgment: N entries listed above.` |

Reason-list composition (comma-separated, in priority order):
1. `after-inventory missing` (capabilityParity = missing)
2. `after-inventory unusable` (capabilityParity = mismatch)
3. `before-inventory missing` (baseline.status = missing)
4. `scan-range mismatch (<detail>)` (scanRangeParity.status = mismatch)
5. `after-inventory incomplete: N file(s) with parse errors` (afterComplete = false)
6. `before-inventory incomplete: N file(s) with parse errors` (beforeComplete = false)

**Why caveated summary matters (P0-2).** `meta.complete === false` means some files were unparseable and their occurrences are NOT in `typeEscapes[]`. Those files are blind spots — a new `any` there could exist and be invisible. Emitting "No silent new any in the scan range" under this condition is a false claim. The caveated line is the honest form.

## 8. Known risks

- **Cold-cache concurrent writes.** `any-inventory.mjs` writes to `<output>/any-inventory.json`. If post-write runs while pre-write's hook is still renaming the file, a race could occur. Mitigation: atomic temp+rename from P2-0's rename step; post-write's spawn runs AFTER checking the advisory is fully written (file exists + JSON.parse succeeds).
- **Advisory file without `preWrite.anyInventoryPath`.** Happens when the P2-0 hook skipped (user ran pre-write with `--no-fresh-audit`) or failed. Delta degrades to `baseline: {status: 'missing'}` → all observed classify as `observed-unbaselined`. No `silent-new` produced — honest downgrade.
- **`any-inventory.mjs` invocation in post-write cold-cache.** `--no-fresh-audit` present → `<output>/any-inventory.json` must already exist from a prior run. Absent case: `capabilityParity: 'missing'`, `entries: []`, `failures: [{kind: 'after-inventory-missing'}]`. Pinning test.
- **Ambiguous planned-match — baseline routing.** When planned intent realistically covers N occurrences but the P2-1 v1 schema has no `count` field, the tiebreak picks one canonically-first candidate and the remainder passes through baseline comparison. If a remainder was pre-existing, it correctly classifies as `pre-existing` and does NOT fire a false `silent-new`. The `ambiguous-planned-match` diagnostic is the audit trail.
- **Scan-range mismatch preserves planned evidence.** Mismatched scan ranges invalidate the before/after comparison only; planned matching stays valid. Unmarked remainders become `observed-unbaselined`; `removed` is not computed; `requiredAcknowledgements: []`.
- **Capability source is the inventory, not the advisory.** P1's `advisory.capabilities` is a symbols projection; type-escape capability lives in `any-inventory.json.meta.supports.typeEscapes` + `escapeKinds[]`.
- **`normalizeCodeShape` export coupling.** §3.3 exception adds one `export` keyword to extract-ts-escapes. Test pins the exported reference is the same function used internally, so a future P2-0 refactor that forks the normalizer breaks the pin and forces re-sync.
- **Parse-error blind zone (P0-2 honest handling).** `afterInventory.meta.complete === false` means some files could not be parsed; their occurrences are NOT in `typeEscapes[]`. Silent-new detection for those files is IMPOSSIBLE this run. P2-1 renders the blind zone explicitly (`inventoryCompleteness` block + caveated summary line). The "No silent new any in the scan range" phrasing is reserved for fully-clean runs. Without this guardrail, a parse-error regression would silently weaken the `no silent new any` invariant.
- **Parent `docs/history/phases/p2/session.md` predates v3.** The strategic parent still describes ambiguity-remainder as hard `silent-new`, a different `computeDelta` signature, and a "4 classification cases" framing. Per §1 Implementation Authority, implementers follow THIS file (`docs/history/phases/p2/p2-1.md` v3). Reconciling `docs/history/phases/p2/session.md` is a separate follow-up ticket.
- **`computeDelta` purity discipline.** The pure-function claim hinges on caller-injected `deltaInvocationId`. A future refactor that moves id generation inside `computeDelta` would silently break test stability (same inputs producing different outputs). The source-grep pinning test (§5.1) catches this structurally.
- **Large inventories.** `O(n log n)` sort + `O(n)` delta via occurrenceKey map. A repo with 10k `any` occurrences would produce a 10k-entry delta; Markdown may be long. P2-1 v1 does NOT truncate (honest full list); a `--top N` flag is a P2+ cosmetic improvement.
