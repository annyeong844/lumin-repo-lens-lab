## 4. Output shapes

### 4.1 `canonical-draft/type-ownership.md`

Format mirrors `docs/spec/SPEC-canon-generator.md` §6 exactly. High-level:

```md
# Type ownership draft

Generated: <ISO>
Scope: <TS/JS including tests | TS/JS production files>
Source: fresh-ast-pass [+ symbols.json when present]

| Name | Identity | Owner | Fan-in | Status | Tags |
|------|----------|-------|-------:|--------|------|
| `<name>` | `<ownerFile>::<exportedName>` | `<ownerFile>:<line>` | <N> | <label marker> | <tags...> |
...

## Notes
- <[확인 불가] entries with reasons>
- <Group-level rows for DUPLICATE_STRONG / DUPLICATE_REVIEW / LOCAL_COMMON_NAME / ANY_COLLISION>
```

**Pinning rules:**
- Every cell goes through `escapeMdCell` (escapes `|`, `\`, newlines).
- Code-style cells (type names, identities, file paths) go through `codeCell` (backtick wrap + inner-backtick escape).
- `<label marker>` is EXACTLY one of the 9 canonical labels from `classification-gates.md §9`.
- Identity strings use `exportedName` (never `typeName` alone — canonical/fact-model §3.1 is explicit).
- `[확인 불가]` rows carry a reason: `ambiguous-star-reexport` / `resolveIdentity-depth-exceeded` / `unresolved-specifier`.

### 4.2 Non-overwrite file versioning (SPEC §5.5)

If `<canon-output>/type-ownership.md` exists from a prior run: emit `<canon-output>/type-ownership.v{N}.md` where `N = max(existing) + 1`. Never overwrite a prior draft.

### 4.3 Existing-canon observational-only header (SPEC §5.4)

If `<root>/canonical/type-ownership.md` exists: the draft's first block is a `> ⚠ Existing canon detected` header per SPEC §5.4 verbatim. No comparison logic runs in Phase 1 — that's Post-P3 (`check-canon.mjs`).

### 4.4 CLI flags + exit codes

```
generate-canon-draft.mjs
  --root <path>                         repository root (required)
  --output <dir>                        pipeline artifact dir (required; reads symbols.json if present)
  --canon-output <dir>                  draft output dir (defaults below)
  --source <type-ownership>             P3-1 supports type-ownership only (other sources → P3-2/3/4)
  --include-tests / --no-include-tests  forwarded to collectFiles
  --production                          opt-out shortcut (equivalent to --no-include-tests)
  --exclude <pattern>                   forwarded to collectFiles (multiple allowed)
```

`--canon-output` default resolution per SPEC §3.1:
- Standalone: `<root>/canonical-draft/`.
- Invoked through `audit-repo.mjs` (P3-4): `<output>/canonical-draft/`.

Exit codes:
- 0 — draft emitted (may contain `[확인 불가]` rows; they are informational).
- 1 — fatal error: missing required flag, unreadable `symbols.json` (when specified), malformed canonical during Step 0 drift-test stage.

Stdout: short status banner + draft path. Stderr: `[canon-draft]` diagnostics.

### 5.2 Step 1 — `_lib/canon-draft.mjs` pure helpers

Additional pins (reviewer P1 — canonical contamination Rule 0 scope):

### 5.3 Step 2 — type-ownership emitter

**Test first:** `tests/test-canon-draft-type-ownership.mjs`. Cases (target ~20):

Identity aggregation:
- Single owner, single consumer → `single-owner-strong` or `-weak` per fan-in.
- Two identities same name, both live → `DUPLICATE_STRONG` or `DUPLICATE_REVIEW` per fan-in.
- Two identities same name, name ∈ `LOW_INFO_NAMES`, low fan-in → `LOCAL_COMMON_NAME`.
- Same-file + cross-file duplicate pairings — identity is `ownerFile::exportedName`, never collapsed on name alone (per `identity-and-alias.md §2`).
- `typeName` ≠ `exportedName` hypothetical pin (matches Step 0 pin from a different angle): identity cell uses `exportedName`.

Re-export chain (P0 reviewer clarification — terminal identity is always the owner, barrier names are record-keeping only):
- `export { X } from './y'` via `symbols.json.reExportsByFile` → resolve to owner; **terminal identity is `src/y.ts::X`**, fan-in attributed there; barrel `src/index.ts` recorded in `reExportedThrough`, NOT as the identity.
- `export { X as Y }` alias hop:
  - Consumer `import { Y } from './index'` matches the barrel record's `exportedName: "Y"`.
  - Chain then rewrites the name-at-barrier back to the source-side `importedName: "X"`.
  - **Terminal identity is the defining owner identity: `src/y.ts::X` — never `src/index.ts::Y`.**
  - `Y` is recorded as a re-export/barrier name in the trace, not as the final owner identity. This is the alias-fidelity rule from `canonical/identity-and-alias.md` §4: `importedName` (source-side) and `localName`/`exportedName` at the alias site (consumer-side) are preserved separately so the identity never collapses to the wrong one.
  - Pinning test: fan-in attributes to `src/y.ts::X` with count 1; `reExportedThrough: ['src/index.ts']`; NO row keyed on `src/index.ts::Y` appears in the draft.
- Star re-export with multi-source name collision → `[확인 불가]` with reason `ambiguous-star-reexport`.
- 8-hop resolve depth exceeded → `[확인 불가]` with reason `resolveIdentity-depth-exceeded`.

Contamination routing:
- `anyContamination.label === 'severely-any-contaminated'` on a single identity → classifier returns `severely-any-contaminated`.
- Group of 2+ all `any-contaminated` → `ANY_COLLISION`.

Markdown rendering:
- Cells with `|` in a type parameter — escaped.
- Identity cell always uses the `codeCell` wrapper + `ownerFile::exportedName` format.
- Label marker in table matches canonical §9 label exactly.

**Impl:** extend `_lib/canon-draft.mjs` with:
- `collectTypeIdentities({ symbols, root, includeTests, exclude, languages })` — returns `{typeDefsByIdentity, typeNameIndex, typeUsesByIdentity, diagnostics}`. Scan-range args thread through to `collectFiles` when `symbols` is absent; when `symbols` is present, the args are cross-checked against `symbols.meta.{scope, includeTests, exclude}` — mismatch surfaces as a diagnostic (not fatal; draft still emits).

## 8. Known risks

- **`symbols.json` absent.** Fresh AST pass replaces it but re-export chain resolution is limited — any barrel with a star re-export or alias rename that we don't have a symbols snapshot for emits `[확인 불가]`. Documented in the meta block (`source: "fresh-ast-pass"` + `barrels-opaque: true`).
- **`LOW_INFO_NAMES` mirror drift.** Step 0 drift test is the guardrail; if it regresses, P3-1 halts before classifier emits any draft. A canonical file edit that adds a new LOW_INFO entry must be followed by a mirror update in the same PR — the Step 0 test is structurally failing otherwise.
- **8-hop resolve depth.** `identity-and-alias.md §6` caps at 8 hops to prevent infinite re-export loops. A deliberately-pathological repo will emit `[확인 불가]` rows with `resolveIdentity-depth-exceeded` reason. This is honest; consumers downgrade the confidence.
- **Star re-export ambiguity.** `export * from './x'` + `export * from './y'` both exporting `Foo` → `identity-and-alias §9` mandates `[확인 불가]` with reason `ambiguous-star-reexport`. P3-1 surfaces these as draft Notes rows, not silent drops.
- **`typeName` vs `exportedName` drift.** Canonical §3.1 says they're equal on `type-owner` facts. A future extractor change that makes them diverge (e.g., namespace-nested exports) would invalidate "read `typeName` for display, `exportedName` for identity" — the code constant `CANON_IDENTITY_FIELD = 'exportedName'` pins the source column so a future divergence is caught by the Step 0 hypothetical-pin test.
- **Non-overwrite versioning race.** Two concurrent `generate-canon-draft.mjs` runs might both pick the same `N` for `type-ownership.v{N}.md`. P3-1 v1 uses a simple `max(existing)+1` scan; dev-flow-acceptable since manual review follows. A future hardening could use an atomic `O_EXCL` open loop — not blocking for v1.
