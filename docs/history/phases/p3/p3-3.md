**`crossEdgeSource` / `classificationConfidence` contract (P0-3):**
- When `topology.json.crossSubmoduleEdges` IS present ‚Üí `crossEdgeSource: "full-list"`, `classificationConfidence: "high"`. Submodule labels in ¬ß1 are load-bearing.
- When the field is ABSENT (e.g., consumer talks to a pre-P3-3-pre `measure-topology.mjs`) ‚Üí `crossEdgeSource: "top-30-only"`, `classificationConfidence: "medium"`. The draft header carries a prominent warning (see ¬ß4.1), and `isolated-submodule` rows are suppressed (replaced with `scoped-submodule` as the conservative default) ‚Äî a zero in/out reading from top-30-only is not strong enough to claim isolation. This is the honest-degradation path: classification only rides on evidence strong enough to support it.

#### 5.3.1 Submodule inventory source order (reviewer P1-4)

The aggregator's `submodulesByPath` Map MUST be populated from the following sources in strict order. This prevents "submodule absent from cross-edges is invisible to inventory" failures ‚Äî an isolated submodule has zero cross-edges by definition but still must appear as a row.

```
Submodule inventory source order (first wins; later steps augment, do not replace):

1. triage.boundaries[]      ‚Äî monorepo-workspaces / monorepo-packages modes.
                              Each boundary contributes one submodule entry.
2. triage.topDirs keys      ‚Äî single-package mode.
                              Each top-level directory contributes one entry.
3. topology.nodes top-dir   ‚Äî fallback when triage.json is absent.
                              For each file key in nodes, extract the first
                              path component; each distinct component is one
                              submodule entry.
4. crossSubmoduleEdges      ‚Äî AUGMENTS in/out-degree on existing entries only.
                              NOT a source of new inventory entries. An edge
                              endpoint that doesn't match any existing
                              inventory entry surfaces as
                              `submodule-boundary-mismatch` diagnostic.
```

Rationale: inventory and edge evidence come from different producers with different coverage. An isolated submodule has zero edges yet exists as a structural unit; losing it would silently drop `isolated-submodule` candidates. Edge evidence refines classification; it never creates inventory.

Test pin: a fixture with `triage.topDirs: {a: ..., b: ..., c: ...}` and `crossSubmoduleEdges: [{from: 'a', to: 'b', count: 3}]` ‚Üí inventory has 3 rows (`a`, `b`, `c`); `c` appears with zero in/out-degree and (in full-list mode) `isolated-submodule` label.

- Row order: submodules sorted alphabetically. Stable output.
- Cells with `|` in submodule path ‚Äî escaped.
- SCC member file cells use `codeCell`.
- `triage.json.mode` surfaced in meta header line.

### ¬ß11.1 Submodule classification

Evaluation order ‚Äî **first match wins**:

| # | Condition | Label | Marker |
|---|---|---|---|
| 0 | submodule has any file in `topology.json.sccs[].members` | `cyclic-submodule` | ‚ùå (canon invariant violation) |
| 1 | `inDegree == 0` AND `outDegree == 0` AND `crossEdgeSource == "full-list"` | `isolated-submodule` | ‚Ñπ (may be dead-weight or entry-points) |
| 2 | `inDegree >= 5` | `shared-submodule` | ‚úÖ (canon-worthy hub) |
| 3 | `outDegree > inDegree` AND `inDegree < 5` | `leaf-submodule` | ‚öÝ (one-way consumer) |
| 4 | else | `scoped-submodule` | ‚Ñπ |

Rule 0 fires FIRST. A cyclic submodule is a canon violation regardless of centrality ‚Äî label it cyclic even at high fan-in. Cycles are the primary flag; hubs / leaves are secondary structural observations.

Rule 1 (`isolated-submodule`) requires `crossEdgeSource == "full-list"` ‚Äî i.e., the consumer read from `topology.json.crossSubmoduleEdges` (full list), not `crossSubmoduleTop` (top-30 display). This is the degraded-mode guard: a zero in/out reading from top-30-only evidence is NOT strong enough to claim isolation. In top-30-only mode, the condition falls through to Rule 4 (`scoped-submodule`) ‚Äî the conservative default.

Thresholds (`5` for Rule 2) are v1 empirical. Per-repo override lives in the canonical file's evolution, not the spec.

`inDegree` / `outDegree` are CROSS-SUBMODULE edge counts ‚Äî internal-to-submodule edges don't count. Derived from `topology.json.crossSubmoduleEdges[]` (full list, required for high-confidence classification).

### ¬ß11.5 Critical notes

**`isolated-submodule` is NOT "dead".** A submodule with zero cross-submodule in-edges and zero out-edges may be an entry point (e.g., `bin/`, `tests/`), a config-only directory (`configs/`), or a genuinely orphaned area. Cross-check with `triage.json.shape.testFiles` and `boundaries` before suggesting removal. Wording must be `isolated-submodule`, never "dead" or "unused".

**`forbidden-cycle` is a canon-invariant violation, not a code-removal command.** Resolution options mirror ¬ß5.3 for types: rename one module, merge two modules, or introduce an abstraction to break the cycle. The skill flags; humans / LLMs decide.

**`oversize` is NOT "split this".** A 500-LOC test file that exists because the suite is exhaustive is legitimate; a 500-LOC production file that grew by accretion is a refactor signal. Canon draft surfaces the observation; reviewer classifies.

### ¬ß11.6 Identity semantics

Submodule identity is a path string. No `ownerFile::exportedName` format. No alias resolution (submodule paths don't alias the way type/helper names do). No LOW_INFO_* list (path-based, not name-based).

SCC member identities ARE file paths (used in the ¬ß3 cycle listing). These files belong to a submodule whose classification is `cyclic-submodule`; the per-file identity is report-level, not classification-level.

### ¬ß11.7 Fan-in evidence source (reviewer P0-3 honest-degradation)

Classification in ¬ß11.1 uses `inDegree` / `outDegree` computed from `topology.json.crossSubmoduleEdges` (the full untruncated cross-submodule edge list, producer field landed in P3-3-pre). This is the only honest source for "zero in/out" claims (Rule 1 `isolated-submodule`).

Consumers reading a pre-P3-3-pre `topology.json` that lacks `crossSubmoduleEdges` degrade to `crossSubmoduleTop` (top 30). In that mode:
- The draft meta records `crossEdgeSource: "top-30-only"` + `classificationConfidence: "medium"`.
- The draft renders a prominent degradation header: "‚öÝ Submodule classification derived from top-30 cross-edge lens. `isolated-submodule` suppressed. Upgrade `measure-topology.mjs` to emit full `crossSubmoduleEdges` for high-confidence classification."
- Rule 1 (`isolated-submodule`) is STRUCTURALLY suppressed (see ¬ß11.1 guard) ‚Äî rows that would match it fall through to Rule 4 (`scoped-submodule`).
- Rules 0 (SCC), 2 (`shared-submodule`), 3 (`leaf-submodule`), 4 (`scoped-submodule`) still fire but with medium confidence.

Classification confidence is part of the draft's honest record ‚Äî the reviewer / LLM consuming the draft knows what the labels are riding on.

### ¬ß11.7 Out of scope for ¬ß11

- Edge-level classification. Cross-submodule edges are REPORTED (¬ß2 of the draft) but not individually classified.
- Boundary-rule enforcement ‚Äî `canonical/fact-model.md ¬ß3.5` declares rules; checking observed edges against rules is Post-P3 (`check-canon.mjs`).
- Workspace-level classification. Monorepo workspaces are LISTED (¬ß5 of the draft) but not individually classified in v1. Future extensions may add `workspace-isolated` / `workspace-interconnected` labels.
- Per-file SCC labeling. Files inside SCCs inherit `cyclic-submodule` at the submodule level; there is no per-file cycle label in v1.
