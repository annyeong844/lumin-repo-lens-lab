Definition-inventory source (reviewer P0-2 decision — fresh AST wins).** The helper **definition** inventory for P3-2 comes from a fresh `extractDefinitionsAndUses` pass over the scan range, not from `call-graph.json.topCallees`. Rationale: `topCallees` is a "called helpers" view — exported helpers that are never called, callback-passed, or reflection-invoked are invisible to it. Definition inventory keyed on `topCallees` would produce a "central-helpers-only" registry that silently omits `zero-internal-fan-in-helper` rows. P3-2 MUST emit those rows.
>
> **PF-4. Fan-in semantics (reviewer P0-3 decision — consumer-file fan-in, derived from AST).** "Fan-in" in P3-2 = `|distinctConsumerFiles|` for a helper identity, NOT aggregated call-site count. A single consumer file calling `parseJson()` 10 times yields fan-in 1, not 10.
>
> `build-call-graph.mjs` today aggregates `count` across call-sites in `topCallees: [{file, name, count}]` — that is CALL-SITE count, not consumer-file count. The `callEdges` array (`{from, to, callee}` triples) is NOT serialized to `call-graph.json`, so P3-2 cannot derive consumer-file fan-in from the current artifact.
>
> P3-2 resolves this by computing consumer-file fan-in directly in the fresh AST pass (per PF-3): while walking consumer files, the pass tracks `distinctConsumerFiles: Map<identity, Set<consumerFile>>`. This avoids both call-site double-counting AND the `callEdges`-not-serialized gap. `call-graph.json` is then optional cross-check only (if `topCallees[identity].count > 0` but our AST pass found 0 consumers, surface as a diagnostic — may indicate reflection/callback invocation we missed). Meta block records `fanInKind: "consumer-file-count"` unambiguously. See §3.1 for the full source chain.

## 4. Output shapes

### 4.1 `canonical-draft/helper-registry.md`

Format mirrors the `type-ownership.md` structure for visual parity — reviewers reading both drafts should recognize the shape instantly. Columns differ because helpers carry a signature and centrality tier where types carry a kind and fan-in.

```md
# Helper registry draft

Generated: <ISO>
Scope: <TS/JS including tests | TS/JS production files>
Source: fresh-ast-pass [+ symbols.json.reExportsByFile when present] [+ call-graph.json cross-check when present]
FanInKind: consumer-file-count
Mode: fresh-ast | fresh-ast + helper-owner enrichment  [fresh-ast-only is the 2026-04-21 default]

| Name | Identity | Owner | Signature | Fan-in | Status | Tags | Any / unknown signal |
|------|----------|-------|-----------|-------:|--------|------|----------------------|
| `<name>` | `<ownerFile>::<exportedName>` | `<ownerFile>:<line>` | `<signature>` | <N> | <label marker> | <tags...> | — or <contamination summary> |
...

## Notes
- <[확인 불가] entries with reasons>
- <Diagnostics: call-graph cross-check mismatches, if any>
- <Group-level rows for HELPER_DUPLICATE_STRONG / HELPER_DUPLICATE_REVIEW / HELPER_LOCAL_COMMON / ANY_COLLISION_HELPER>
```

**Pinning rules** (mirrors P3-1 §4.1):

- Every cell goes through `escapeMdCell`.
- Code-style cells (names, identities, file paths, signatures) go through `codeCell`.
- `<label marker>` is EXACTLY one of the helper labels from `canonical/classification-gates.md` §10 (P3-2-pre prerequisite).
- Identity strings use `exportedName` (never the raw callee name if it differs from the export — alias fidelity per `identity-and-alias.md` §4).
- **Fan-in is consumer-file-count** per PF-4 (NOT call-site count). Meta block records `fanInKind: "consumer-file-count"` unconditionally.
- `[확인 불가]` rows carry one of four reasons (locked in Step 0 extension — §5.1): `ambiguous-star-reexport` / `resolveIdentity-depth-exceeded` / `unresolved-specifier` / `helper-owner-facts-unavailable` (new reason — replaces `call-graph-missing`; used when contamination facts are absent AND a row's label would materially depend on them — see §4.5 policy).
- Signature column carries the inferred type signature — from `symbols.json.helperOwnersByIdentity.signature` when Option A enrichment is active; falls back to the extractor's raw arity string otherwise (e.g., `(a, b, c) => <unknown>`). Empty signature is rendered as `—`, never `()` which would imply zero-arg incorrectly.

### 4.5 Meta block + contamination policy

Analogous to the P3-1 meta JSON block. Additional fields for P3-2:

```json
{
  "tool": "generate-canon-draft.mjs",
  "schemaVersion": 1,
  "target": "helper-registry",
  "mode": "fresh-ast" | "fresh-ast + helper-owner enrichment",
  "fanInKind": "consumer-file-count",
  "helperContamination": "available" | "unavailable",
  "filesScanned": {N},
  "identitiesFound": {N},
  "inputs": {
    "symbols.json.reExportsByFile": {"present": true | false},
    "symbols.json.helperOwnersByIdentity": {"present": true | false},
    "call-graph.json": {"present": true, "mode": "cross-check-only", "staleness": "fresh" | "stale" | "absent"}
  },
  "thresholds": {
    "centralTier": 3,
    "sharedTier": 1,
    "staleCallGraphHours": 24,
    "lowInfoHelperNames": [...]
  }
}
```

**Contamination policy (reviewer P1-6).** Two orthogonal decisions:

1. **Row-level rendering.** When `helperContamination === "unavailable"` (i.e., `symbols.json.helperOwnersByIdentity` absent — fresh-AST-only mode), the `Any / unknown signal` column renders `—` on EVERY row. No `"unavailable"` string in cells — the table stays clean; readers consult meta for the limitation reason. When contamination IS available, the column renders the label + key measurements exactly like P3-1 does (`severely-any-contaminated (anyParams 3/5, asAnyCount 4)`).
2. **Label availability.** The `severely-any-contaminated-helper` label is producible ONLY when contamination is available. In fresh-AST-only mode, the classifier treats every identity as clean (contamination argument = undefined) and the label is structurally unreachable. This is PINNED by a dedicated assertion in Step 0 extension and Step 1: pass the classifier `helperContamination === 'unavailable'` context, run it across 50+ fixture identities, assert `severely-any-contaminated-helper` never appears. Similarly `ANY_COLLISION_HELPER` (Rule 0 — universal contamination quantifier) is unreachable in fresh-AST-only mode.

This is the honest degradation path: without producer-side facts, we cannot claim contamination; the draft says so in meta without fabricating row-level summaries.

### 4.6 Staleness threshold

`STALE_CALL_GRAPH_HOURS = 24` — a module-level constant in `_lib/canon-draft.mjs` (parallel to `LOW_INFO_NAMES` / `LOW_INFO_HELPER_NAMES`). Reuses the fact-model §4 "< 1 day" confidence downgrade threshold. When `call-graph.json.meta.generated` is older than 24h measured against the CLI invocation's `Date.now()`:

- Draft header carries `> ⚠ Source call-graph.json is N hours stale. Cross-check diagnostics may under-report.` (rendered with actual hour count, not literal N).
- Meta block records `inputs["call-graph.json"].staleness: "stale"`.
- No other behavior change — classification is unaffected because fan-in already comes from the fresh AST pass, not the stale artifact.

Test pin in Step 4 integration: fixture with a `call-graph.json` whose `meta.generated` is set to 25h ago → draft header contains "stale" warning.

### Proposal for `canonical/classification-gates.md` §10 — Helper classification

A **group** is the set of helper identities sharing an `exportedName`. Fan-in per identity is computed from `call-graph.json.topCallees` (or `symbols.json.helperOwnersByIdentity[identity].fanIn` when the producer-emitted facts are available).

**§10.1 Helper group classification** (parallel to §2 type group classification) — first match wins. Fan-in is consumer-file-count (PF-4) — the count of distinct consumer files that import the helper, not aggregated call-site count.

| # | Condition | Label | Marker |
|---|---|---|---|
| 0 | `size ≥ 2` AND EVERY member's `anyContamination.label ∈ {any-contaminated, severely-any-contaminated}` | `ANY_COLLISION_HELPER` | ⚠ every-helper-contaminated |
| 1 | `size ≥ 2` AND max-fanIn ≥ 3 AND sum-fanIn ≥ 3 | `HELPER_DUPLICATE_STRONG` | ⚠ needs-resolution |
| 2 | `size ≥ 2` AND `name ∈ LOW_INFO_HELPER_NAMES` (§10.4) AND max-fanIn < 3 | `HELPER_LOCAL_COMMON` | ℹ context-local |
| 3 | `size ≥ 2` (any other case) | `HELPER_DUPLICATE_REVIEW` | ⚠ review-needed |

**§10.2 Helper single-identity classification** (parallel to §4) — first match wins:

| # | Condition | Label | Marker |
|---|---|---|---|
| 0 | `anyContamination.label === 'severely-any-contaminated'` | `severely-any-contaminated-helper` | ⚠ severely-contaminated |
| 1 | `name ∈ LOW_INFO_HELPER_NAMES` AND `fanIn < 3` | `low-signal-helper-name` | ℹ low-signal |
| 2 | `fanIn ≥ 3` | `central-helper` | ✅ canon-candidate |
| 3 | `fanIn ∈ {1, 2}` | `shared-helper` | ℹ weak-fan-in |
| 4 | `fanIn === 0` | `zero-internal-fan-in-helper` | ℹ may-be-public-api |

Parallel to type §4 precedence: Rule 0 contamination wins over everything; Rule 1 `low-signal-helper-name` wins over the fan-in tiers when fan-in is below the centrality threshold. A helper named `get` with fan-in 2 is `low-signal-helper-name`, not `shared-helper`. A helper named `get` with fan-in 3 is `central-helper` (Rule 2 wins over Rule 1 when fan-in ≥ 3) — mirrors the `Result` + high-fanIn pattern from type §2.

**§10.3 Label set (drift-locked)** — the canonical 9-label helper set:

```
HELPER_DUPLICATE_STRONG
HELPER_DUPLICATE_REVIEW
HELPER_LOCAL_COMMON
ANY_COLLISION_HELPER
severely-any-contaminated-helper
central-helper
shared-helper
zero-internal-fan-in-helper
low-signal-helper-name
```

Emitted labels MUST be a subset of this list. `tests/test-classification-gates.mjs` Step 0 extension pins both the mirror and the emitted set.

**§10.4 `LOW_INFO_HELPER_NAMES` (reviewer P1-5 — distinct list).** Helpers use a separate low-info list from types because the semantic weight differs — `Props` / `Options` / `Result` are type-ish low-info names; `get` / `parse` / `format` are helper-ish low-info names. Mixing them in one list would force the drift-test to carry false positives in each direction.

Initial list (kept small per reviewer guidance — ~10-15 to start, grow empirically):

```
get
set
parse
format
fetch
load
save
build
make
create
update
handle
run
process
convert
```

Mirror in `_lib/canon-draft.mjs` as `LOW_INFO_HELPER_NAMES`, drift-locked by Step 0 extension against this §10.4 list (byte-equal, same names same order). Per-repo override config follows the same roadmap as `LOW_INFO_NAMES` (canonical-file-level evolution, not P3-2 spec).

Note: the existing §3 `LOW_INFO_NAMES` is NOT changed. Helpers that happen to carry a type-ish name (rare — `Result()` as a helper is already `parseJson`-ish) still use the helper list; helper classification never reads `LOW_INFO_NAMES`.

**§10.5 Rule 0 scope.** Universal quantifier, not existential — parallel to §2 Rule 0. A mixed group (one `any-contaminated` + one clean) is NOT `ANY_COLLISION_HELPER`; it falls through to Rule 1/2/3 per fan-in math. Rationale is identical to the type case: a partially-contaminated group is a normal duplicate with a warning tag, not an opaque-blob collision.

**§10.6 Fan-in semantics.** `fanIn = |distinctConsumerFiles|`. A single consumer file calling a helper 10 times yields fan-in 1. This differs from `call-graph.json.topCallees.count` which aggregates call-site count. P3-2 emits fan-in only via the AST pass described in docs/history/phases/p3/p3-2.md §3.1; `call-graph.json` is cross-check only.
