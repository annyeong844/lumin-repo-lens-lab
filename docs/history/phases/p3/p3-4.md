> **PF-2. Identity shape — COHORT-keyed for classification, with per-item identities split file-vs-symbol (reviewer P1-11).** P3-4's naming canon operates on **cohorts** — groups of items sharing a dimension where a convention can be measured. Two cohort kinds in v1:
> - **File-naming cohorts**: keyed on submodule path (e.g., `_lib`, `src/components`). Members are files in that submodule; the "convention" is the **normalized file-basename casing** (see file basename normalization rules below).
> - **Symbol-naming cohorts**: keyed on `(submodule, kind)` pairs (e.g., `_lib::type-export`, `src/components::helper-export`). Members are exported symbols of that kind; the "convention" is the exported-name casing.
>
> Identity string format:
> - **Cohort identity**: `<submodule>` for file cohorts; `<submodule>::<kind>` for symbol cohorts.
> - **Per-item identity** (appears in §3 outlier listings, not in classification): `<ownerFile>` for file items; `<ownerFile>::<exportedName>` for symbol items. The two shapes are structurally distinct so §3 rows can be read without ambiguity about which cohort they belong to.
>
> `kind` ∈ `{type-export, helper-export, constant-export}`. Classification rules (reviewer P0-7):
> - **`type-export`**: AST kind ∈ `{TSInterfaceDeclaration, TSTypeAliasDeclaration, TSEnumDeclaration, TSModuleDeclaration}` (same set as P3-1 `TYPE_OWNER_KINDS`).
> - **`helper-export`**: `FunctionDeclaration` AND exported `const`/`let`/`var` whose initializer is a `FunctionExpression` or `ArrowFunctionExpression`. JSX components (`export const UserCard = () => <div/>`) fall here in v1 — see risk N-3.
> - **`constant-export`**: exported `const`/`let`/`var` whose initializer is NOT a function expression (literals, object/array literals, calls returning values, class instances).
>
> Note: the extractor emits `const-var`/`let-var`/`var-var` without distinguishing arrow vs literal initializer (P3-2 HW-9 limitation). v1 uses a shallow re-check — if the AST node's `declarations[i].init.type` is `ArrowFunctionExpression` or `FunctionExpression`, the export goes into `helper-export`; else `constant-export`. This is a thin parse on top of `extractDefinitionsAndUses` return values; documented in §5.2 impl block.
>
> **File basename normalization rules (reviewer P0-2 — locks dogfood expectations).** Before `detectConvention(name)` runs on a file path, strip:
> 1. The final extension (`.mjs`, `.ts`, `.tsx`, `.js`, `.jsx`, `.cjs`, `.mts`, `.cts`, `.d.ts`).
> 2. Common multi-part suffixes BEFORE the final extension: `.test`, `.spec`, `.stories`, `.d`.
>
> Examples:
> - `_lib/canon-draft.mjs` → `canon-draft` → `kebab-case` ✓
> - `src/components/UserCard.tsx` → `UserCard` → `PascalCase` ✓
> - `tests/user-profile.test.tsx` → `user-profile` → `kebab-case` ✓
> - `src/api.d.ts` → `api` → `camelCase` (single lowercase segment) ✓
>
> Normalization is applied to the basename ONLY — the submodule path (cohort identity) is not normalized.
>

> **File-naming cohorts ALWAYS use `collectFiles(root, ...)`** — they need every file regardless of whether the file exports anything. A file named `helpers.js` with no exports still contributes to the `kebab-case-dominant` vs `PascalCase-dominant` file cohort observation.

### 4.5 Orchestrator — `audit-repo.mjs --canon-draft`

```
--canon-draft                      Opt-in, NOT in quick/full/ci profiles.
--sources <csv>                    Optional. Default: all CANON_DRAFT_SOURCES.

```

**Source universe constant (reviewer P1-9).** Exported from `_lib/canon-draft.mjs`:

```js
export const CANON_DRAFT_SOURCES = Object.freeze([
  'type-ownership',
  'helper-registry',
  'topology',
  'naming',

```

`--sources <csv>` values are validated against this constant. Unknown values → exit 1 with stderr listing all four valid names.

**`perSource` is requested-only** (reviewer P1-9). When the user specifies `--sources type-ownership,naming`, `manifest.canonDraft.perSource` has exactly two keys. Sources not requested do NOT appear as skipped entries — their absence is the signal.

`manifest.canonDraft` shape:

```json
{
  "requested": true,
  "ran": true,
  "requestedSources": ["type-ownership", "helper-registry", "topology", "naming"],
  "perSource": {
    "type-ownership":  { "ran": true,  "exitCode": 0, "draftPath": "<root>/canonical-draft/type-ownership.md" },
    "helper-registry": { "ran": true,  "exitCode": 0, "draftPath": "<root>/canonical-draft/helper-registry.md" },
    "topology":        { "ran": false, "exitCode": 2, "reason": "topology.json absent — run measure-topology.mjs first" },
    "naming":          { "ran": true,  "exitCode": 0, "draftPath": "<root>/canonical-draft/naming.md" }
  },
  "draftPaths": [
    "<root>/canonical-draft/type-ownership.md",
    "<root>/canonical-draft/helper-registry.md",
    "<root>/canonical-draft/naming.md"
  ]
}
```

**`manifest.canonDraft.ran` definition (reviewer P1-10).**

> `ran = true` iff at least one requested source exited 0 AND emitted a draftPath. If zero requested sources succeeded, `ran = false` and `reason: 'all requested sources failed'`.

This is the single source-of-truth definition. Per-source outcome lives in `perSource.<name>.ran` (boolean per source). Orchestrator `ran` is the disjunction across requested sources.

**Orchestrator exit code contract:**
- 0 — `manifest.canonDraft.ran === true`. Every requested source ran with exit 0, OR some sources degraded with exit 1/2 but at least one draft was emitted. `--canon-draft` is advisory by default (parallels `--post-write` default).
- 1 — invocation error (unknown `--sources` value) OR `manifest.canonDraft.ran === false` (every requested source failed).
- 2 — propagated from a source ONLY if `--strict-canon-draft` is set (analog to `--strict-post-write`). v1 does NOT implement `--strict-canon-draft`; `--canon-draft` is always advisory. Flagged as a Post-P3-4 hardening option.

**Profile exclusion:** neither `quick` nor `full` nor `ci` profiles include `--canon-draft`. The flag is deliberate-invocation-only; CI should opt in explicitly.

  - Edge cases: empty string, single char, leading digit, underscores only.
- `normalizeFileBasename(filePath)` (canonical §12.6 — P0-2):
  - `'_lib/canon-draft.mjs'` → `'canon-draft'` → `kebab-case`.
  - `'src/UserCard.tsx'` → `'UserCard'` → `PascalCase`.
  - `'tests/user-profile.test.tsx'` → `'user-profile'` → `kebab-case`.
  - `'src/api.d.ts'` → `'api'` → `camelCase`.
  - `'src/Foo.stories.tsx'` → `'Foo'` → `PascalCase`.
  - Pin: normalization applied to basename only; submodule path untouched.
- `classifyNamingCohort`:
  - Cohort of 10 names, 9 `camelCase` + 1 `PascalCase` → `camelCase-dominant` + consistencyRate 0.9.
  - Cohort of 10 names, 5/5 split → `mixed-convention`.
  - Cohort of 2 names (< 3 threshold) → `insufficient-evidence`.
  - **Effective-size pin (P0-3):** Cohort of 10 names where 8 are low-info → effectiveMembers = 2 → `insufficient-evidence`; outlier detection suppressed.
  - **Effective-size pin (P0-3) passing case:** Cohort of 10 names where 7 are non-low-info → effectiveMembers = 7 → dominance fires normally.
  - Cohort with all low-info names → `insufficient-evidence`.
  - Cohort of 10 names, 7 camelCase + 2 PascalCase + 1 UPPER_SNAKE → `camelCase-dominant` (majority threshold met).
- `classifyNamingItem` (rule order pin — P0-4):
  - **Rule 0 priority:** Item name ∈ LOW_INFO → `low-info-excluded` even if cohort is `camelCase-dominant` and the item's own convention matches.
  - **Rule 0 priority in mixed cohort:** Item name ∈ LOW_INFO AND cohort is `mixed-convention` → `low-info-excluded` (Rule 0 wins over Rule 1's "no dominant to deviate from" fallback).
  - **Rule 0 priority in insufficient cohort:** Item name ∈ LOW_INFO AND cohort is `insufficient-evidence` → `low-info-excluded`.
  - Item matches dominant (not low-info) → `convention-match`.
  - Item differs from dominant (not low-info) → `convention-outlier`.
  - Cohort-level `insufficient-evidence`, item not low-info → `convention-match` (Rule 1 fires after Rule 0 miss).
  - Cohort-level `mixed-convention`, item not low-info → `convention-match`.

## 8. Known risks

- **N-1: Cohort selection is a design choice.** "Per-submodule" file cohort vs "per-directory" file cohort changes results. v1 uses submodule resolver (same as P3-3). If a future repo has a convention that doesn't align with submodule boundaries (e.g., `src/components/*` is PascalCase but `src/*` overall is kebab-case), the rolled-up cohort might flag the mix as `mixed-convention` when per-subdirectory it'd be clean. Documented limitation; future work may add sub-cohort drilling.
- **N-2: Consistency threshold is empirical.** "Majority dominance" threshold v1: ≥ 60% of non-low-info members share the convention. Below 60% → `mixed-convention`. Canonical §12 is authoritative; adjust empirically.
- **N-3: JSX components naming conflict.** React components SHOULD be PascalCase (both file + export); utility functions in `.tsx` files SHOULD be camelCase. v1 treats all .tsx exports as ordinary type/helper exports — a file cohort of mostly-component .tsx files might emit `PascalCase-dominant` which reflects the component pattern. Accept this as honest; future work may introduce a JSX-component sub-cohort.
- **N-4: `symbols.json` enrichment vs fresh AST coherence.** If symbols.json was produced against a different scan-range than our current CLI flags, the two sources could give inconsistent answers. Meta block records which was used; reviewer cross-checks.
- **N-5: Low-info exclusion can hide genuine outliers.** `Result` (low-info) in a camelCase helper cohort is excluded from the dominance count — but if someone exports a helper named `Result`, it would never show as an outlier. Canonical §12 accepts this as designed; low-info names are semantically conventional-unenforceable.
- **N-6: Orchestrator exit code 2 propagation.** Topology source emits exit 2 when producer artifact missing. Orchestrator v1 treats this as a soft failure (advisory path). If CI wants exit-2 propagation, `--strict-canon-draft` is the future hardening.
- **N-7: Non-overwrite versioning race** — same as P3-1/2/3.
- **N-8: `check-canon.mjs` not in P3** — existing-canon case emits observational header only.
- **N-9: Per-source spawn cost.** Orchestrator runs 4 CLI invocations serially. v1 acceptable (canon drafting is a deliberate step, not hot-path). Future optimization could parallelize, but each source opens oxc bindings which have startup cost — serial is simpler.

### §12. Naming classification

Parallel to §2/§4 (types), §10 (helpers), §11 (topology), adapted for naming conventions. Consumed by the P3-4 canon draft generator (`canonical-draft/naming.md`) per `docs/history/phases/p3/p3-4.md` v1.

**Identity for naming** is **cohort-keyed** — NOT `ownerFile::exportedName` (per-item) and NOT submodule path (that's topology).
- **File-naming cohort identity**: `<submodule>` (e.g., `_lib`).
- **Symbol-naming cohort identity**: `<submodule>::<kind>` where `kind ∈ {type-export, helper-export, constant-export}`.

Per-item identities (`ownerFile::exportedName`) appear inside §3 outlier listings but are NOT primary for §12.1 classification.

### §12.1 Cohort classification

A **cohort** is a set of items (files or exported symbols) sharing a submodule (+ kind). Cohort membership computation uses EFFECTIVE size — raw members minus low-info names (`LOW_INFO_NAMES` §3 + `LOW_INFO_HELPER_NAMES` §10.4) per reviewer P0-3.

Evaluation — **compute dominance on effective members, then match**:

1. For each raw member, compute the observed convention:
   - **File cohort member**: apply basename normalization first (strip file extension `.mjs`/`.ts`/`.tsx`/`.js`/`.jsx`/`.cjs`/`.mts`/`.cts`/`.d.ts` and multi-part suffixes `.test`/`.spec`/`.stories`/`.d` — see p3-4 PF-2), then `detectConvention(normalizedBasename)`.
   - **Symbol cohort member**: `detectConvention(exportedName)` directly.
2. Strip low-info names from dominance computation. `effectiveMembers = members.filter(m => !lowInfoExclusions.has(m.name))`.
3. If `effectiveMembers.length < 3` → `insufficient-evidence` (outlier detection suppressed).
4. Else: `dominance = max(count per convention) / effectiveMembers.length`.
5. If `dominance >= 0.6` (60% majority threshold) → `<convention>-dominant`.
6. Else → `mixed-convention`.

| # | Condition | Label | Marker |
|---|---|---|---|
| 0 | `effectiveMembers.length < 3` | `insufficient-evidence` | ℹ |
| 1 | dominance ≥ 0.6 AND convention = camelCase | `camelCase-dominant` | ✅ |
| 1 | dominance ≥ 0.6 AND convention = PascalCase | `PascalCase-dominant` | ✅ |
| 1 | dominance ≥ 0.6 AND convention = kebab-case | `kebab-case-dominant` | ✅ |
| 1 | dominance ≥ 0.6 AND convention = snake_case | `snake_case-dominant` | ✅ |
| 1 | dominance ≥ 0.6 AND convention = UPPER_SNAKE | `UPPER_SNAKE-dominant` | ✅ |
| 2 | dominance < 0.6 | `mixed-convention` | ⚠ |

### §12.2 Per-item classification

For each item in a cohort, apply rules in order — **first match wins** (reviewer P0-4, `low-info-excluded` always has item-level priority):

| # | Condition | Label | Marker |
|---|---|---|---|
| 0 | name ∈ LOW_INFO_NAMES ∪ LOW_INFO_HELPER_NAMES | `low-info-excluded` | ℹ |
| 1 | cohort is `insufficient-evidence` OR `mixed-convention` | `convention-match` | ✅ (no dominant to deviate from) |
| 2 | item convention === cohort dominant convention | `convention-match` | ✅ |
| 3 | item convention ≠ cohort dominant convention | `convention-outlier` | ⚠ |

Rule 0 fires FIRST regardless of cohort state. Rationale: a low-info member of a `mixed-convention` cohort is STILL conceptually excluded from enforcement — the item's own characterization is "this name shouldn't count" independently of how the cohort classifies overall. The §3 Outliers table surfaces `low-info-excluded` rows for transparency so reviewers can see which members were skipped from dominance math.

Insufficient-evidence and mixed-convention cohorts then suppress outlier detection (Rule 1) — non-low-info items get `convention-match` because there's no clear dominant to deviate from.

### §12.4 Critical notes

**`mixed-convention` is NOT "inconsistent".** A mixed cohort may reflect a deliberate split (domain types PascalCase, protobuf-generated types snake_case) — observation, not verdict.

**`convention-outlier` is NOT "rename this".** Outliers may be intentional (legacy compat, external API mirroring). Canon drafts surface observations; reviewer decides.

**`insufficient-evidence` is the honest-small-cohort label.** A single-file submodule cannot sustain a convention claim. Not a defect.

**`low-info-excluded` is semantic design, not an oversight.** `get` / `parse` / `Result` carry convention meaning poorly because they're universal. Excluding them prevents skewed dominance counts.

### §12.5 Conventions enum (NAMING_CONVENTIONS constant)

Mirrored in `_lib/canon-draft.mjs`. Used by `detectConvention(name)` return values.

```
camelCase       — starts lowercase, later segments capitalize: fooBar, getUser
PascalCase      — starts uppercase: FooBar, User
kebab-case      — hyphen-separated lowercase: foo-bar, user-service
snake_case      — underscore-separated lowercase: foo_bar, user_service
UPPER_SNAKE     — underscore-separated uppercase: FOO_BAR, MAX_RETRY
mixed           — doesn't match any single pattern above (e.g., `foo_Bar`, `Foo-bar`)
```

### §12.6 File basename normalization (reviewer P0-2)

File cohort members are file-basenames, not full paths. Before `detectConvention(name)` runs on a file cohort member, the name MUST be normalized:

1. Strip the final language extension: `.mjs` / `.ts` / `.tsx` / `.js` / `.jsx` / `.cjs` / `.mts` / `.cts` / `.d.ts`.
2. Strip common multi-part suffixes that appear BEFORE the final extension: `.test` / `.spec` / `.stories` / `.d`.

Reference examples:

| File path | Normalized basename | detectConvention result |
|-----------|---------------------|-------------------------|
| `_lib/canon-draft.mjs` | `canon-draft` | `kebab-case` |
| `src/components/UserCard.tsx` | `UserCard` | `PascalCase` |
| `tests/user-profile.test.tsx` | `user-profile` | `kebab-case` |
| `src/api.d.ts` | `api` | `camelCase` |
| `src/legacy_module.js` | `legacy_module` | `snake_case` |
| `src/FOO.test.mjs` | `FOO` | `UPPER_SNAKE` |

Normalization applies to basename only — the submodule path (cohort identity) is never normalized. This is a deterministic string transform; `_lib/canon-draft.mjs::collectNamingCohorts` applies it before each `detectConvention` call.

### §12.8 Per-item identity shape (reviewer P1-11)

The Outliers table (§3) surfaces per-item rows. The Identity column uses a structurally distinct shape depending on whether the item is a file or a symbol, so a reader can tell at a glance which cohort kind the row belongs to:

| Cohort kind | Per-item identity shape | Example |
