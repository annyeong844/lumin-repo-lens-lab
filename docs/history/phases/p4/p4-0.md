# docs/history/phases/p4/p4-0.md - Shape-Hash Foundation Closure

> **Phase:** P4 v1 - shape-hash evidence foundation.
> **Role:** provide grounded structural-shape evidence for supported exported
> object/interface types without changing existing duplicate-name labels.
> **Status:** v1 closed (2026-04-22). The original planning session has been
> implemented, checkpoint-reviewed, and hardened against the P4 review HIGH
> findings.
> **Last updated:** 2026-04-22

---

## 0. Closure Rule

P4 v1 is complete as an evidence layer, not as a TypeScript semantic checker.

Allowed claim:

```text
shape-hash evidence is available for supported exported object/interface
types when shape-index.json exists or when pre-write cold-cache builds it.
```

Not allowed:

```text
DUPLICATE_STRONG means structurally identical.
```

Not allowed:

```text
P4 can safely merge duplicate types automatically.
```

The implementation must continue to reject fuzzy field overlap, grep-style
shape matching, and best-effort duplicate merging. Structural equality claims
must come from a validated `shape-index.json` fact, not from ad hoc matching.

## 2. Current Truth

`DUPLICATE_STRONG` still means:

```text
same exportedName group
AND max(fanIn) >= 3
AND sum(fanIn) >= 3
```

It still does not mean:

```text
the duplicate type declarations have identical structure
```

P4 v1 adds a separate evidence channel:

```text
same-shape evidence      -> duplicate identities share a validated shape hash
different-shape evidence -> duplicate identities have validated distinct hashes
partial evidence         -> only some identities have shape facts
degraded evidence        -> shape-index.json is incomplete
generated-only summary   -> generated-file duplicate groups are summarized
```

These notes enrich review. They do not rewrite the canonical §9 label contract.

## 3. Implemented Scope

### 3.1 P4-1 Shape Hash Core

Implemented in `_lib/shape-hash.mjs`.

Supported v1 declarations:

- exported `interface` declarations;
- exported `type` aliases whose right-hand side is an object type literal;
- local declarations exported via `export { Local as Public }`.

Normalized hash-bearing inputs:

- normalized schema version;
- object shape kind;
- fields sorted by field name/kind/stable payload;
- field name;
- optional flag;
- readonly flag;
- normalized type text;
- method signature marker for supported methods.

Unsupported shapes emit diagnostics, not fake facts.

Important safety pin: same-file declaration merging is not partially hashed.
If multiple declarations resolve to the same `ownerFile::exportedName`,
P4 v1 emits `declaration-merge-unsupported` and produces no fact for that
identity.

Deferred semantic cases:

- checker-grade declaration merging;
- generic instantiation;
- mapped types;
- conditional types;
- interface `extends` expansion;
- imported type alias resolution;
- type intersection/union equivalence beyond token-normalized field text.

### 3.2 P4-2 Shape Index Producer

Implemented by `build-shape-index.mjs` and
`_lib/shape-index-artifact.mjs`.

Artifact:

```text
<output>/shape-index.json
```

Minimum shape:

```json
{
  "schemaVersion": "shape-index.v1",
  "meta": {
    "tool": "build-shape-index.mjs",
    "source": "fresh-ast-pass",
    "scope": "TS/JS production files, exported types only",
    "observedAt": "...",
    "complete": true
  },
  "facts": [],
  "groupsByHash": {},
  "diagnostics": []
}
```

Each fact carries canonical metadata: `source`, `scope`, `confidence`, and
`observedAt`.

Generated-file evidence is tagged on facts when a strong generated signal is
present (for example `routeTree.gen.ts`, `.gen.ts`, `generated/`, or a generated
header comment). Facts are not dropped; consumers may choose to summarize them.

### 3.3 P4-3 Shared Shape Index Contract

Implemented in `_lib/shape-index-schema.mjs`.

Consumers must validate the artifact through the shared parser. The parser
checks:

- `schemaVersion === "shape-index.v1"`;
- `facts[]` exists;
- `groupsByHash` exists;
- each fact has one valid `sha256:<64hex>` hash;
- optional `generatedFile` metadata has the exact
  `{ kind: "generated-file", source: "path"|"header", evidence: string }`
  shape;
- duplicate fact identities are rejected;
- `groupsByHash` exactly matches the identities represented by `facts[]`.

This prevents ghost matches where `groupsByHash` names an identity that has no
corresponding fact, and prevents malformed generated-file metadata from hiding
hand-authored evidence in summarized draft output.

### 3.4 P4-4 Pre-Write Shape Lookup

Implemented in `_lib/pre-write-lookup-shape.mjs`.

Grounded `SHAPE_MATCH` is allowed only when:

- `shape-index.json` exists and validates through the shared parser;
- intent supplies an exact `shape.hash`, or a `shape.typeLiteral` that can be
  normalized by the same P4 normalizer;
- matching facts exist in validated `facts[]`.

Legacy `{ fields: [...] }` entries remain display hints only. Field names alone
are not structural equality evidence.

When `shape-index.json` is incomplete, positive matches remain grounded, but
absence claims degrade to `UNAVAILABLE`.

### 3.5 P4-5 P3 Type-Ownership Enrichment

Implemented in `_lib/canon-draft-types.mjs` and `generate-canon-draft.mjs`.

`generate-canon-draft.mjs --source type-ownership` loads optional
`shape-index.json`. The canonical type-ownership table remains unchanged, so
P5 parser contracts stay stable. Shape evidence renders as Markdown bullets in
`## Shape evidence`.

The renderer can surface:

- `same-shape evidence`;
- `different-shape evidence`;
- `shape evidence partial`;
- `shape evidence degraded`.

Generated-only duplicate groups are summarized instead of expanded into long
per-hash bullets. This keeps generated route/type artifacts from drowning out
reviewable hand-authored duplicate-shape evidence while preserving the raw facts
in `shape-index.json`.

## 4. Cold-Cache Integration

`pre-write.mjs` conditionally requests `shape-index.json` from the cold-cache
preflight when any intent shape carries `hash` or `typeLiteral`.

This keeps ordinary pre-write latency unchanged while making exact shape
lookups work without a manual producer step.

`--no-fresh-audit` still disables all cold-cache producers, including
`build-shape-index.mjs`.

## 5. Acceptance Gates

P4 v1 is pinned by tests covering:

- equal object/interface fields with different field order -> same hash;
- different field type -> different hash;
- optional vs required field -> different hash;
- readonly vs mutable field -> different hash;
- unsupported mapped/generic shape -> diagnostic, no fake fact;
- parse error in owner file -> no high-confidence fact;
- declaration merging -> `declaration-merge-unsupported`, no partial fact;
- malformed index -> pre-write shape lookup `UNAVAILABLE`;
- ghost `groupsByHash` entry without `facts[]` support -> `UNAVAILABLE`;
- missing index -> pre-write shape lookup `UNAVAILABLE`;
- exact hash / `typeLiteral` match -> `SHAPE_MATCH`;
- incomplete index miss -> `UNAVAILABLE`;
- complete index miss -> `NOT_OBSERVED`;
- generated-file shape facts remain present but generated-only duplicate notes
  are summarized;
- no heuristic fallback path in pre-write shape lookup;
- type-ownership draft table remains canon-drift parser compatible.

## 6. Marketplace Claim Boundary

Allowed after P4 v1:

```text
Supported exported object/interface types can carry grounded shape-hash
evidence.
```

Allowed:

```text
Duplicate type-name groups may be enriched with same-shape or different-shape
evidence when the shape index has facts for the identities.
```

Still not allowed:

```text
All duplicate types are semantically identical.
```

Still not allowed:

```text
P4 can safely merge duplicate types automatically.
```

Shape equality is evidence for review. It is not, by itself, a migration plan.

## 7. Closure Decision

P4 v1 should stop here.

Further work such as checker-grade declaration merging, generic expansion,
mapped type evaluation, and public API semantic precision belongs in P6, where
P6-0 can measure whether precision changes reduce false positives and whether
runtime remains acceptable.

Recommended next step:

```text
P6-0 dogfood / measurement on a pinned corpus.
```

Then proceed to P6-1+ only after a measurement artifact exists.
