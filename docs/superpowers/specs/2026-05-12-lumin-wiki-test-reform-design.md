# Lumin Wiki And Test Reform Design

## Context

Lumin Repo Lens has grown through many small regression-driven PRs. That
worked for fast recovery, but the accumulated design knowledge now lives across
conversation history, `docs/spec`, `docs/lab`, generated test docs, and test
fixtures. Future work should not require reconstructing that context from chat.

This design introduces a maintainer wiki and a test reform contract. The first
slice is documentation-only: it creates the knowledge structure and rules for
future test refactors. It does not move test files, change test behavior, or
modify analyzer logic.

## Goals

- Create a persistent maintainer wiki that compounds project knowledge instead
  of rediscovering it from raw docs each session.
- Make the wiki a synthesis layer over existing source-of-truth artifacts, not a
  replacement for specs, tests, or lab evidence.
- Define the test reform direction before editing test suites.
- Shift future TDD from "fails because a helper does not exist" toward "fails
  because a real edge case is not handled."
- Keep future test migrations small, reversible, and evidence-grounded.

## Non-Goals

- Do not move or rename test suites in this slice.
- Do not add a test runner, fixture harness, or new analyzer behavior in this
  slice.
- Do not treat wiki pages as release artifacts or public skill documentation.
- Do not replace `tests/README.md`; it remains generated by
  `scripts/update-test-doc.mjs`.

## Wiki Architecture

The wiki lives under `docs/lumin-wiki/` and has three layers:

- `AGENTS.md`: maintainer rules for wiki edits.
- `index.md` and `log.md`: navigation and chronological change record.
- Topic pages under `overview.md`, `workstreams/`, and `concepts/`.

Raw sources remain authoritative:

- specs in `docs/spec/`
- lab evidence in `docs/lab/`
- implementation code under root scripts and `_lib/`
- public skill code under `skills/lumin-repo-lens-lab/`
- generated test inventory in `tests/README.md`
- individual test suites in `tests/test-*.mjs`

Wiki pages synthesize these sources and must link back to them when making
claims.

## Test Reform Contract

Future test work should classify each suite by the risk it protects:

- Component contract: a focused helper or library unit.
- Artifact shape: JSON/Markdown schema, counters, support flags, and mirrors.
- Resolver or reachability blind zone: unsupported, candidate, or scoped blocker
  behavior.
- Regression edge case: a real bug class such as prototype-name methods,
  unreachable SCCs, namespace re-export fan-in, or `import.meta.glob`.
- Public install or corpus verification: package-installed behavior and
  dogfood evidence.
- Performance measurement: counters, wall time, cache hit/miss rates, and
  variance notes.

Happy-path-only tests are allowed only as smoke tests. Any suite that protects
correctness should include at least one negative or edge-case assertion that
would have failed before the relevant fix.

## TDD Contract

Preferred red tests fail because existing behavior is wrong on a concrete
fixture:

- prototype-named class methods must not crash dictionary grouping
- identical tiny exact clones must group when their exact hash matches
- entry-unreachable SCCs must surface as review evidence, not export proof
- unresolved generated/output/dynamic imports must become scoped diagnostics,
  not fake resolved edges
- suppressed pre-write candidates must remain muted until calibrated

Weak red tests fail only because a function, file, or export has not been
created yet. These are acceptable only when introducing a new pure unit with a
clear edge-case fixture in the same test.

## Migration Plan

1. Land this documentation-only scaffold.
2. Inventory suites into the wiki without moving files.
3. Pick one workstream, starting with pre-write or resolver, and tag its suites
   by risk type.
4. Merge duplicate fixture shapes only when the shared fixture still explains
   the original bug.
5. Convert happy-path assertions into edge-case or negative assertions where the
   original behavior was under-specified.
6. Only then consider file movement or harness extraction.

## Acceptance Criteria

- `docs/lumin-wiki/` exists with maintainer edit rules, an index, a log, an
  overview, workstream pages, and concept pages.
- The docs map points maintainers to the wiki without making it a public
  surface.
- The wiki states that raw specs, tests, and lab evidence remain authoritative.
- The test reform page defines risk categories and TDD quality rules.
- No analyzer behavior or test execution behavior changes in this slice.

## Review Notes

This is intentionally a foundation PR. If a future implementation needs to move
tests or consolidate fixtures, it should first update the wiki with the target
suite, original bug, protected invariant, and migration result.
