# Rust Framework Resource Surfaces Producer Design

## Goal

Move `framework-resource-surfaces.json` artifact construction from JS into
`lumin-audit-core` while keeping the existing JS producer script as a thin
compatibility wrapper for file discovery and artifact writing.

This is the next Rust-owned producer slice after `unused-deps`. It should reduce
JS-owned classification logic without taking over source walking or package
discovery policy in the same change.

## Current Owner Split

Current JS ownership:

- `build-framework-resource-surfaces.mjs` discovers candidate files, collects
  `.hbs` resources, reads sampled JS content, discovers package records, calls
  `_lib/framework-resource-surfaces.mjs`, and writes the artifact.
- `_lib/framework-resource-surfaces.mjs` owns artifact construction and
  classification policy:
  - Storybook story files
  - Strapi filesystem API files
  - generated declaration files
  - bundled/minified/vendor/Emscripten build artifacts
  - scaffold templates
  - codemod/testfixture resources

Rust already consumes this artifact through `artifact_summaries.rs` and
manifest evidence projection. The missing piece is moving the producer-side
classification contract into audit-core.

## Proposed Slice

Rust owns only deterministic artifact construction.

JS keeps:

- `collectFiles(...)` and scan/exclude/test policy
- workspace and package discovery through `detectRepoMode(...)`
- `.hbs` filesystem resource discovery
- candidate source content sampling
- final `framework-resource-surfaces.json` file write and console summary

Rust owns:

- request schema validation
- relative path normalization
- package root sorting and nearest package selection
- package-relative path matching
- dependency evidence extraction
- all framework/resource lane classification
- deterministic sorting and artifact-local summary projection

This mirrors the successful `unused-deps` migration shape: JS supplies already
discovered inputs; Rust produces the artifact.

## Rust CLI

Add:

```text
lumin-audit-core framework-resource-surfaces-artifact --input <path|-> [--result-output <path>]
```

The command accepts a JSON request:

```json
{
  "schemaVersion": "lumin-framework-resource-surfaces-producer-request.v1",
  "root": "C:/repo",
  "files": ["src/Button.stories.tsx", "templates/component.hbs"],
  "packageRecords": [
    {
      "root": "C:/repo",
      "relRoot": ".",
      "packageJson": {
        "devDependencies": { "@storybook/react": "^8.0.0" }
      }
    }
  ],
  "contentsByFile": {
    "src/emscripten-bindings.js": "// @ts-nocheck\nGenerated by Emscripten"
  }
}
```

The response is the existing artifact shape:

```json
{
  "schemaVersion": "framework-resource-surfaces.v1",
  "policyVersion": "framework-resource-surface-policy-v1",
  "root": "C:/repo",
  "files": [],
  "summary": {}
}
```

Direct stdout remains supported for small debugging calls. JS wrappers should use
`--result-output` to avoid stdout buffer risk as a normal migration pattern.

## Shared Rust Producer CLI Contract

Producer commands must accept `--input <path|->` and may write either to stdout
or to `--result-output <path>`. JS wrappers must use `--result-output` for
normal repository runs.

Rust must write artifact JSON only to the selected result channel. Diagnostics
must go to stderr. Invalid JSON, schema mismatch, invalid normalized paths, and
failed result-file writes must exit non-zero and must not produce a partial
success artifact.

Wrappers must treat a non-zero exit, missing result file, or malformed result
JSON as producer failure rather than falling back to JS classification.

## Path Normalization Contract

The Rust request boundary is intentionally narrower than repo scanning:

- `files` entries are repo-relative paths supplied by the JS discovery layer.
- backslashes normalize to `/`.
- leading `./` is removed.
- empty or `.` package `relRoot` means the repo-root package.
- package `relRoot` values normalize by trimming trailing `/`.
- `files` entries that are absolute paths, empty paths, or contain `..`
  traversal are invalid request input.
- package and file matching remains case-sensitive.

This keeps discovery policy in JS while making the Rust-owned package matching
deterministic and cross-platform.

## Classification Parity

Rust must preserve the checked JS policy exactly:

- `*.stories.[cm][jt]sx?` creates a `framework-dispatch-entry` lane for
  `framework.storybook`. The current JS policy recognizes `.stories.*`, not
  `.story.*`; this slice must preserve that boundary.
- `src/api/*/{controllers,routes,services}/**` creates a
  `framework-dispatch-entry` lane for `framework.strapi`.
- Storybook/Strapi confidence is `grounded` when the nearest package declares
  matching dependencies; otherwise it is `path-shaped-review`.
- Storybook grounding uses explicit dependency names ported from the JS helper:
  `storybook` and any package starting with `@storybook/`.
- Strapi grounding uses explicit dependency names ported from the JS helper:
  `strapi` and `@strapi/strapi`.
- `**/generated/**/*.d.ts` creates `generated-declaration-surface`.
- `vendor.js`, `*.bundle.*`, `*.min.*`, and sampled Emscripten headers create
  `bundled-build-artifact`.
- `templates/**` and `*.hbs` create `scaffold-template-resource`.
- `resources/codemods/**`, `codemods/**`, and `__testfixtures__/**` create
  `codemod-resource`.

Lane evidence, confidence, reason, default action, and
`affectsAbsenceClaims` must match the existing JS helper output.

Missing sampled content in `contentsByFile` is not an error. It only prevents
content-based evidence such as Emscripten header detection; path-only bundle
rules still apply.

## Sorting And Determinism

The Rust artifact must keep JS ordering contracts:

- package records sort by deepest `relRoot` first, then path
- output files sort by normalized file path
- lanes sort by `lane`, then `reason`
- lane evidence sorts by `kind`, then `field`, then `matched`
- summary maps are stable and sorted lexicographically
- `summary.topExamples` is capped to the first 10 output files

## JS Wrapper Changes

`_lib/framework-resource-surfaces.mjs` becomes a compatibility bridge that calls
Rust. It should keep only:

- exported schema/policy constants
- exported `classifyFrameworkResourceSurfaces(request)` wrapper

It must not keep a parallel JS classifier. Framework dependency tables,
classification order, reason strings, confidence policy, and lane construction
must exist only in Rust after this slice. The direct helper behavior moves to
Rust tests.

`build-framework-resource-surfaces.mjs` should:

- keep current file and package discovery
- build the Rust request
- call `runAuditCoreJsonResultFile(["framework-resource-surfaces-artifact", "--input", "-"], ...)`
- keep the existing artifact write and console summary

## Orchestration Ownership

After the wrapper conversion, `build-framework-resource-surfaces.mjs` should be
marked `ProducerOwner::Rust` in the audit-core orchestration plan. The step name
and script name remain unchanged to preserve manifest and user-facing contract.

## Tests

Rust tests should cover the product behavior currently checked by JS helper
tests:

- golden fixture parity with canonical JSON equivalence, including ordering
- grounded Storybook and Strapi dispatch surfaces
- path-shaped review confidence when framework dependencies are absent
- generated declarations, bundles, templates, codemods, and testfixtures
- `.stories.*` is recognized while `.story.*` remains outside this checked
  policy
- missing `contentsByFile` samples do not hard-stop and do not create
  Emscripten header evidence
- deterministic ordering
- CLI `--result-output`

Node compatibility tests should keep only:

- `build-framework-resource-surfaces.mjs` writes the artifact
- `audit-repo` runs the producer and manifest/summary consume it

Vitest mirrors should be narrowed so they no longer import JS classifier helpers.
They are reference coverage only during this migration and are not the
authoritative gate.

## Non-Goals

This slice does not move:

- `collectFiles(...)`
- scan/exclude/include-tests policy
- workspace discovery
- package manager interpretation
- content sampling policy beyond consuming `contentsByFile`
- manifest summary projection, which is already Rust-owned and separate from
  the artifact-local `summary` this producer constructs

Moving file discovery would couple this small producer migration to scan policy
parity and blind-zone behavior. That is the wrong risk bundle for this step.

## Acceptance Criteria

- `framework-resource-surfaces.json` remains byte-shape compatible with the
  existing checked behavior.
- JS no longer owns framework/resource classification logic.
- The audit orchestration plan reports `build-framework-resource-surfaces.mjs`
  as Rust-owned.
- Focused Rust tests, Node compatibility test, audit-runtime Cargo gate, and
  clippy pass.
