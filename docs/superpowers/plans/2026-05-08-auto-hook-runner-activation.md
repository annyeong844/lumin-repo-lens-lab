# Auto Hook Runner Activation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Activate Phase 1 auto hooks by shipping concrete hook runner scripts and a non-empty `hooks/hooks.json`, while preserving the all-hooks-exit-0 warn-only contract.

**Architecture:** Runner scripts live under plugin-root `hooks/` because the public package already stages `hooks/` and intentionally rejects root `scripts/`. Each runner is a thin JSON-stdin adapter over existing tested core modules: PreToolUse captures preimages and drains reminders, PostToolBatch calls `processPostWriteLite`, Stop observes ACK lines, and UserPromptSubmit drains pending reminders. The runner utility dynamically resolves engine modules from the maintainer checkout (`_lib/`) or from the packaged plugin root (`skills/lumin-repo-lens-lab/_engine/lib/`).

**Tech Stack:** Node.js ESM, Claude Code hook manifest JSON, `_lib/hook-*` core modules, direct Node test scripts, generated skill/plugin packaging.

---

## File Structure

- Create `hooks/_runner-utils.mjs`
  - Owns stdin JSON parsing, best-effort output emission, and dynamic engine module resolution.
- Create `hooks/pre-tool-use.mjs`
  - Captures preimage for `Edit`/`Write`/`MultiEdit`, then drains due reminders as `PreToolUse`.
- Create `hooks/post-tool-batch.mjs`
  - Calls `processPostWriteLite(payload)`, emits returned hook output if present.
- Create `hooks/stop.mjs`
  - Calls `observeStopAcknowledgements(...)` for `AUDIT_ACK` lines.
- Create `hooks/user-prompt-submit.mjs`
  - Drains due reminders as `UserPromptSubmit`.
- Modify `hooks/hooks.json`
  - Add `UserPromptSubmit`, `PreToolUse`, `PostToolBatch`, and `Stop` command entries.
- Modify `tests/test-hook-doctor.mjs`
  - Expect active hook events instead of an empty manifest.
- Create `tests/test-hook-runner-scripts.mjs`
  - Exercise the runner scripts through child processes and JSON stdin.
- Modify `tests/test-plugin-package.mjs`
  - Assert runner scripts ship inside the plugin-root `hooks/` directory.
- Modify `scripts/publish-public-plugin.mjs`
  - Sync and validate `hooks/` when publishing the generated plugin package to the public repo.
- Modify `tests/test-publish-public-plugin.mjs`
  - Assert public dry-run and push flows include the hook manifest and runner scripts.
- Modify `scripts/update-test-doc.mjs`
  - Add the new runner-script test description.
- Generated by `npm run build:skill`
  - No root hook files are mirrored into skill internals; the plugin package copies `hooks/` directly.

- [ ] **Step 1: Implement `pre-tool-use.mjs`**

Algorithm:

```js
const payload = await readJsonFromStdin();
if (!payload) return;
const auditRoot = resolveAuditRoot(payload.cwd ?? process.cwd());
const sid = safeSessionId(payload);
if (auditRoot && tool is mutating and safe path ok) capturePreimage(...);
const drain = auditRoot ? drainDueEventReminders(auditRoot, sid, { hookEventName: 'PreToolUse' }) : null;
emitHookOutput(drain?.output);
```

## Self-Review

- Hook activation is warn-only and all runner scripts exit `0`.
- The plugin root uses `hooks/`, not root `scripts/`, so public package guardrails stay intact.
- No runner persists raw source text or raw hook payload content.
- The runners are thin adapters over already-tested core modules; hook behavior remains testable without Claude Code.
- Manifest activation is explicit and covered by tests.
